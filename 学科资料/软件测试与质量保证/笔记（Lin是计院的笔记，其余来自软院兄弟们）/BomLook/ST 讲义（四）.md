# ST 讲义（四）

# 一、整体脉络：按 V 模型理解四大测试

这份资料的主线是 **V 模型中的四个测试层级**：

> **单元测试 → 集成测试 → 系统测试 → 验收测试**

它们的核心区别可以按以下逻辑记忆：

| 测试阶段 | 测试对象                 | 测试依据                 | 测试方法    | 主要人员          | 关键词                                          |
| -------- | ------------------------ | ------------------------ | ----------- | ----------------- | ----------------------------------------------- |
| 单元测试 | 函数、类、模块等最小单元 | 详细设计                 | 白盒为主    | 开发者            | Unit、White-box、Stub、Driver、XUnit            |
| 集成测试 | 模块之间的接口与协作     | 架构设计                 | 黑盒 + 白盒 | 开发者 + 测试人员 | Integration、Interface、Top-down、Bottom-up     |
| 系统测试 | 完整系统                 | 软件需求                 | 黑盒        | 专业测试人员      | System、Requirement、Functional、Non-functional |
| 验收测试 | 面向用户交付的完整系统   | 业务需求、合同、验收标准 | 黑盒        | 用户/客户主导     | Acceptance、Validation、Alpha、Beta             |

------

# 二、Ch7 单元测试 Unit Testing

## 1. 定义

**单元测试**是针对软件中最小可测试单元进行的测试，例如函数、方法、类或模块。

它通常由开发人员编写，属于 **白盒测试**，重点关注程序内部逻辑、异常处理、边界条件、数据处理和代码质量。

关键词：

> White-box：单元测试在充分了解实现细节的情况下编写
> Developers：单元测试由你，开发者，与实现同时编写。
> Small Units：单元测试应一次分离出一个软件
> Verification：确保你构建了“正确的软件”
> Early Error Detection

------

## 2. 单元测试的作用

单元测试的价值可以概括为四点：

1. **尽早发现错误**
   缺陷越早发现，修复成本越低。
2. **帮助定位错误**
   如果单元测试失败，通常说明问题集中在该单元内部。
3. **提高代码质量**
   促使开发者写出结构清晰、可测试、低耦合的代码。
4. **为集成测试打基础**
   在模块集成前保证每个单元的最低质量。

------

## 3. 单元测试的核心组件

单元测试环境中常见三个组成部分：

| 组件   | 中文     | 作用                                       |
| ------ | -------- | ------------------------------------------ |
| Unit   | 被测单元 | 要验证的函数、类或模块                     |
| Stub   | 桩模块   | 替代被测单元依赖的下层模块                 |
| Driver | 驱动模块 | 替代上层调用者，调用被测单元并传入测试数据 |

记忆方式：

> **Driver 调用 Unit，Unit 调用 Stub。**

也就是说：

```text
Driver → Unit → Stub
```

Driver 控制测试输入和调用流程；Stub 模拟依赖模块的行为。

------

## 4. 单元测试常见内容

单元测试通常覆盖：

| 测试内容 | 示例                                   |
| -------- | -------------------------------------- |
| 正常输入 | 输入合法参数，检查输出是否正确         |
| 异常输入 | 空值、非法值、错误类型                 |
| 边界条件 | 最大值、最小值、零值、数组边界         |
| 分支路径 | if/else、循环、异常分支                |
| 依赖隔离 | 使用 Stub、Mock 隔离数据库、网络接口等 |

------

## 5. FIRST 原则

单元测试应满足 FIRST 原则：

| 字母 | 含义            | 解释               |
| ---- | --------------- | ------------------ |
| F    | Fast            | 测试执行要快       |
| I    | Isolated        | 测试之间相互独立   |
| R    | Repeatable      | 结果可重复         |
| S    | Self-validating | 自动判断通过或失败 |
| T    | Timely          | 测试应及时编写     |

------

## 6. XUnit 与 TDD

### XUnit

XUnit 是一类自动化单元测试框架，不同语言有不同实现：

| 语言       | 常见框架          |
| ---------- | ----------------- |
| Java       | JUnit             |
| Python     | pytest / unittest |
| JavaScript | Jest              |
| C#         | NUnit / xUnit.net |

### TDD

TDD 即测试驱动开发，基本思想是：

```text
先写测试 → 再写代码 → 通过测试 → 重构
```

核心是“只写足够让测试通过的代码”。

------

# 三、Ch8 集成测试 Integration Testing

## 1. 定义

**集成测试**是在单元测试之后，将多个模块组合起来，测试模块之间的接口、调用关系和数据传递是否正确。以确保系统的组件或子系统能够正确协同工作。测试用例将探索组件间的不同相互作用，确保产生正确的结果。

它既可以使用黑盒方法，也可以使用白盒方法。

关键词：

> Interface
> Module Interaction
> Driver
> Stub
> Top-down
> Bottom-up
> Sandwich

------

## 2. 集成测试关注点

集成测试主要检查：

1. 模块之间参数传递是否正确
2. 接口调用是否符合设计
3. 数据流是否正确
4. 模块组合后是否产生新的错误
5. 单个模块正确时，组合后是否仍然正确

------

## 3. Drivers and Stubs

集成测试中也会使用 Driver 和 Stub （这两部分是临时的软件组件）：

| 组件   | 作用                             |
| ------ | -------------------------------- |
| Driver | 模拟上层模块，调用被测模块       |
| Stub   | 模拟下层模块，返回固定或有限结果 |

在集成测试中，Stub 和 Driver 的开发成本可能较高，这是集成测试的重要缺点之一。

------

## 4. 集成测试方法（基于分解的方法）

基于系统的功能特征进行分解，功能特征由模块所做的事情定义，即模块执行的动作或活动。

### 4.1 Big Bang Integration

一次性集成所有模块，然后整体测试。

优点：

- 简单直接
- 不需要复杂的增量集成计划

缺点：

- 错误定位困难
- 一旦失败，很难判断是哪一个模块或接口出错

------

### 4.2 Top-down Integration

从主控模块开始，逐步向下集成。

```text
Main → 上层模块 → 中层模块 → 底层模块
```

![image-20260517094233314](https://cdn.jsdelivr.net/gh/BomLook/blog-pic@main/img/image-20260517094233314.png)

特点：

- 需要大量 Stub 模拟下层模块，且 Stubs 必须始终返回正确的值。向下移动一层，更新 Stubs。
- 可以较早验证系统主流程
- 适合主控逻辑明确的系统
- 任何故障都必须出现在新集成的单元中
- 一次性代码编程

------

### 4.3 Bottom-up Integration

![image-20260517094950569](https://cdn.jsdelivr.net/gh/BomLook/blog-pic@main/img/image-20260517094950569.png)

从底层模块开始，逐步向上集成。

```text
底层模块 → 中层模块 → 上层模块 → Main
```

特点：

- 需要 Driver 调用底层模块
- 底层模块可以较早得到充分测试
- 主流程较晚才能完整验证
- 任何故障（很可能）都出在新集成的代码中
- 代码可以被重复利用

------

### 4.4 Sandwich Integration

结合 Top-down 和 Bottom-up。

特点：

- 上层和底层同时推进
- 中间层最后汇合
- 可以减少单纯 Top-down 或 Bottom-up 的局限
- 需要的 Stub 和 driver 更少

------

## 5. 集成测试方法对比

| 方法      | 优点             | 缺点                            |
| --------- | ---------------- | ------------------------------- |
| Big Bang  | 简单，准备工作少 | 错误定位困难                    |
| Top-down  | 早期验证主流程   | 需要大量 Stub                   |
| Bottom-up | 底层模块测试充分 | 需要大量 Driver，主流程较晚验证 |
| Sandwich  | 兼顾上下层       | 计划和协调更复杂                |

------

## 6. 集成测试优缺点

### 优点

- 直观清晰
- 可以逐步构建系统
- 有助于发现模块接口问题
- 使用已验证模块进行集成，风险较低
- 故障隔离随集成单元数量而异

### 缺点

- Stub 和 Driver 开发成本较高
- 某些功能分解结构可能与真实接口不完全对应
- 集成顺序设计不当会影响测试效率

------

# 四、Ch9 系统测试 System Testing

## 1. 定义

**系统测试**是在完整系统层面进行的测试，用来检查软件是否满足顶层需求。前提是假设所有组件都正常运行。

它主要使用 **黑盒测试技术**，测试依据来自软件需求和具体用例。

关键词：

> System
> Requirement
> Black-box
> Functional Testing：确保符合功能需求
> Non-functional Testing：确保符合非功能需求
> Regression Testing

------

## 2. 系统测试类型

系统测试按测试目的可分为：

```text
系统测试
├── 功能测试
├── 非功能测试
├── 回归测试
└── 性能测试
```

------

# 五、功能测试 Functional Testing

![image-20260517101501200](https://cdn.jsdelivr.net/gh/BomLook/blog-pic@main/img/image-20260517101501200.png)

## 1. 定义

功能测试用于验证软件行为是否符合功能需求。默认的测试。

它不仅测试正常输入，也测试错误输入，以验证系统是否能合理处理异常情况。

------

## 2. 功能测试常用方法

功能测试主要使用黑盒测试技术：

| 方法       | 作用                       |
| ---------- | -------------------------- |
| 等价类划分 | 将输入划分为有效类和无效类 |
| 边界值分析 | 测试边界附近的数据         |
| 决策表测试 | 处理多条件组合             |
| 场景测试   | 基于真实业务流程设计测试   |
| 错误推测   | 根据经验猜测容易出错的位置 |
| 随机测试   | 使用随机数据发现隐藏问题‘  |

与单元测试类似，测试用例和测试数据是通过所用技术进行选择的。

唯一的区别是，与其调用指定参数的方法，不如根据需要将数据输入接口，然后从界面收集结果。

# 七、回归测试 Regression Testing

## 1. 定义

回归测试用于确认代码修改没有破坏已有功能、性能、安全性或其他质量属性。

典型场景：

- 修复 Bug 后
- 新增功能后
- 重构代码后
- 修改接口后
- 系统升级后

------

## 2. 回归测试本质

回归测试的本质是：

> 代码发生变化后，重新运行相关测试用例。

如果有良好的自动化测试体系，回归测试可以快速、稳定、重复执行。

------

# 八、测试自动化 Test Automation

## 1. 优点

| 优点           | 说明                       |
| -------------- | -------------------------- |
| 提高效率       | 重复性测试可以自动执行     |
| 降低人工错误   | 自动化结果更稳定           |
| 支持大规模测试 | 可覆盖多环境、多配置       |
| 支持并行执行   | 多平台同时运行             |
| 支持无人值守   | 可定时执行                 |
| 长期 ROI 高    | 初期成本高，但长期节省成本 |

------

## 2. 缺点

| 缺点           | 说明                               |
| -------------- | ---------------------------------- |
| 初始成本高     | 工具、脚本、维护都需要投入         |
| 需要编程能力   | 测试人员需具备技术基础             |
| 不适合所有测试 | 用户体验、视觉感受等难以完全自动化 |
| 维护成本存在   | 软件变化后脚本也要更新             |

------

# 九、性能测试 Performance Testing

## 1. 定义

性能测试用于检查系统在一定工作负载下的响应性、稳定性、可扩展性和资源使用情况。

核心目标：

1. 发现性能瓶颈
2. 验证响应时间是否满足需求
3. 验证系统在不同负载下是否稳定
4. 判断系统最大承载能力

------

## 2. 性能测试关注指标

| 指标            | 含义         |
| --------------- | ------------ |
| Processor Usage | CPU 使用率   |
| Memory Use      | 内存使用情况 |
| Disk Time       | 磁盘读写时间 |
| Bandwidth       | 网络带宽     |
| Response Time   | 响应时间     |
| Throughput      | 吞吐量       |
| Hits per Second | 每秒命中数   |

------

## 3. 性能测试类型

| 类型                       | 关注点                         |
| -------------------------- | ------------------------------ |
| 负载测试 Load Testing      | 系统在预期用户负载下的表现     |
| 压力测试 Stress Testing    | 系统在极端负载下的表现和崩溃点 |
| 持久测试 Endurance Testing | 系统长时间运行是否稳定         |
| 尖峰测试 Spike Testing     | 系统面对突发流量峰值的反应     |
| 容量测试 Volume Testing    | 系统面对大量数据时的表现       |

------

## 4. 性能测试、负载测试、压力测试、容量测试对比

| 类型     | 核心对象       | 目标                         | 负载特点                     |
| -------- | -------------- | ---------------------------- | ---------------------------- |
| 性能测试 | 整体性能表现   | 建立性能基准，发现瓶颈       | 覆盖多种负载                 |
| 负载测试 | 大量用户       | 验证预期负载下是否正常       | 通常低于崩溃阈值             |
| 压力测试 | 极限用户或数据 | 找到系统崩溃点和恢复能力     | 高于正常负载，接近或超过极限 |
| 容量测试 | 大量数据       | 检查数据规模变化对系统的影响 | 数据量大                     |

一句话记忆：

> **负载测试看“能不能扛住预期用户”，压力测试看“什么时候崩”，容量测试看“数据多了会怎样”，性能测试是总称。**

------

# 十、Ch11 验收测试 Acceptance Testing

## 1. 定义

验收测试是软件测试的最终阶段，用于判断系统是否满足业务需求和交付标准。

它通常在系统测试之后、正式发布之前进行。

关键词：

> Acceptance
> Business Requirement
> Customer
> End User
> Black-box
> Validation

------

## 2. 验收测试特点

| 项目     | 内容                       |
| -------- | -------------------------- |
| 测试阶段 | 系统测试之后，发布之前     |
| 测试对象 | 完整系统                   |
| 测试依据 | 业务需求、合同、验收标准   |
| 测试方法 | 黑盒测试                   |
| 主导人员 | 客户或最终用户             |
| 核心目的 | 判断产品是否可以交付或发布 |

------

## 3. 验收测试组成

验收测试通常包括：

# 十一、Alpha 测试与 Beta 测试

## 1. Alpha Testing

Alpha 测试是验收测试的一种，通常在开发方环境中进行。

特点：

- 开发者视角
- 内部或受控环境
- 测试活动可控
- 问题通常由开发团队高优先级修复
- 参与者可能是内部测试人员、技术专家或少量潜在用户

------

## 2. Beta Testing

Beta 测试通常在真实用户环境中进行。

特点：

- 用户视角
- 真实市场环境
- 测试活动不可完全控制
- 问题多以用户反馈或建议形式出现
- 通常发生在 Alpha 测试之后

------

## 3. Alpha 与 Beta 对比

| 维度     | Alpha Testing                    | Beta Testing         |
| -------- | -------------------------------- | -------------------- |
| 阶段     | 客户确认第一步                   | 客户确认第二步       |
| 视角     | 开发者视角                       | 用户视角             |
| 环境     | 受控测试环境                     | 真实市场环境         |
| 控制程度 | 可控                             | 不可控               |
| 参与者   | 内部人员、技术专家、专业测试人员 | 目标用户             |
| 问题处理 | 直接记录并高优先级修复           | 作为后续版本改进依据 |
| 测试周期 | 多轮                             | 通常 1-2 轮          |

------

| 维度             | 单元测试                  | 集成测试                                | 系统测试                   | 验收测试                   |
| ---------------- | ------------------------- | --------------------------------------- | -------------------------- | -------------------------- |

| 测试人员         | 开发人员                  | 开发人员 + 测试人员                     | 专业测试人员               | 用户/客户主导              |

| 常见技术         | XUnit、Stub、Driver、Mock | Big Bang、Top-down、Bottom-up、Sandwich | 功能、非功能、回归、性能   | Alpha、Beta、配置审查      |

# 十三、推荐笔记脉络分析线

# 十四、最适合考试记忆的压缩版

如果你要整理成一页复习笔记，可以压缩成下面这版：

```text
单元测试：
测最小代码单元，白盒，开发者负责。
重点：Stub、Driver、XUnit、FIRST、TDD。
目标：早发现错误、定位错误、提高代码质量。

集成测试：
测模块之间的接口和协作，黑盒+白盒。
重点：Big Bang、Top-down、Bottom-up、Sandwich。
目标：发现接口错误和模块组合错误。

系统测试：
测完整系统，黑盒，依据软件需求。
包括：功能测试、非功能测试、回归测试、性能测试。
性能测试包括：负载、压力、持久、尖峰、容量测试。

验收测试：
测交付系统，黑盒，用户/客户主导。
依据：业务需求、合同、验收标准。
包括：配置审查、有效性测试、Alpha、Beta。

四大测试顺序：
Unit → Integration → System → Acceptance

测试对象变化：
代码单元 → 模块接口 → 完整系统 → 用户验收

测试依据变化：
详细设计 → 架构设计 → 软件需求 → 业务需求/合同

测试人员变化：
开发者 → 开发者+测试人员 → 专业测试人员 → 用户/客户
```
