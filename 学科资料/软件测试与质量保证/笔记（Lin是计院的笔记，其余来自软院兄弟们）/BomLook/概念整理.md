概念整理
===
## **一、 基础与核心概念**
1.  **Software Testing (软件测试)**
    *   **解释**：一种==系统性的方法==，用于==判断软件质量并发现错误==。它是通过执行软件来**发现缺陷、评估质量并验证软件是否满足预期要求**的过程。
2.  **Software (软件)**

    - 解释：一个软件系统通常由多个部分组成，包括：完成所需功能的**指令**、使程序能够充分操作信息的**数据结构**、用于设置程序的**配置文件**、描述系统结构的**系统文档**，以及解释如何使用系统的**用户文档**。
3.  **PIE Model (PIE模型)**

    - 解释：描述故障导致失效必须经历的三个条件的模型：

      - **Execution (执行/触达)**：程序中包含故障的位置必须被执行到。

      - **Infection (感染)**：执行故障后，程序的状态必须变得不正确（产生Error）。

      - **Propagation (传播)**：感染后的错误状态必须传播到程序的输出，导致输出不正确（产生Failure）。
4.  **Latent Bugs (潜在错误)**

    - **解释**：软件中存在但尚未被发现的错误。

5.  **Validation & Verification (确认与验证)**
    *   **解释**：
        *   **Verification (验证)**：确定**开发过程**某一阶段的产品是否满足上一阶段建立的需求。强调“过程正确性”，即“我们是否正确地构建了产品？”。
        *   **Validation (确认)**：在**软件开发结束时**评估软件，以确保符合预期用途。强调“结果正确性”，即“我们构建的是正确的产品吗？”。
6.  **Fault, Error, Failure (故障、错误、失效)**
    *   **解释**：
        *   **Fault (故障/缺陷)**：软件中的**静态缺陷，是引发问题的根源**（如代码逻辑错误）。
        *   **Error (错误)**：软件运行时因故障被激活而产生的**内部状态异常**（如变量值错误）。
        *   **Failure (失效)**：错误传播到外部，导致**软件表现出与需求不符的错误行为**（如程序崩溃或输出错误结果）。
    *   **关系**：Fault → (执行触发) → Error → (传播) → Failure。存在测试执行故障但不产生错误，或产生错误但未导致失效的情况。
7.  **Specifications (规范)**

    - 解释：软件规范在测试中起着关键作用，它提供了软件正确行为的详细描述。规范必须描述软件的正常行为和错误行为。
8.  **Specifications & Bugs (规范与缺陷)**
    *   **解释**：缺陷与规范的五种关系：
        1.  软件未做规范要求它做的事。
        2.  软件做了规范禁止它做的事。
        3.  软件做了规范未提及的事。
        4.  软件未做产品规范未提及但理应做的事。
        5.  软件难于理解、难于使用或运行缓慢。

## **二、 测试模型与方法**
9.  **Static Whitebox Testing (静态白盒测试)**
    *   **解释**：在**不执行代码**的情况下，通过分析、审查程序结构（如源代码）来发现缺陷的测试方法，属于静态验证。
7.  **White Box Testing (白盒测试)**
    *   **解释**：基于软件的内部实现（代码结构、逻辑路径）来设计和执行测试。测试用例旨在“演习”程序的实现，确保代码的某些方面被测试到。白盒测试关注“实现的覆盖”，但无法发现规范中规定而实现中缺失的功能。
8.  **Black Box Testing (黑盒测试)**
    *   **解释**：完全基于软件的规格说明来设计和执行测试，不涉及代码内部结构。测试用例旨在“演习”软件的规范，验证程序是否满足指定的功能需求。黑盒测试关注“规范的覆盖”，但无法发现实现中存在但规范未要求的额外功能。
9.  **TDD (Test-Driven Development, 测试驱动开发)**
    *   **解释**：一种开发实践，要求先编写自动化测试用例，再编写能通过该测试的代码，最后重构代码。核心流程：编写测试 → 运行测试（失败）→ 编写代码 → 运行测试（通过）→ 重构。它能确保测试被编写，使接口更清晰，并为重构提供信心。
10.  **Static Verification vs. Dynamic Verification (静态验证与动态验证)**
     解释：
     - **Static Verification (静态验证)**：不需要执行代码的验证方法。主要通过直接通读代码，进行静态分析、代码审查、根据编码标准和指南检查等技术来发现问题。
     - **Dynamic Verification (动态验证)**：需要执行代码的验证方法。通过创建测试用例来指导选择合适的测试数据（包含输入值和预期输出值），执行程序后将实际输出与预期输出进行比较来发现问题。
11.  **V Model (V模型)**
     *   **解释**：瀑布模型的扩展，在开发阶段的每个阶段旁都对应一个测试阶段（如需求分析对验收测试，设计对系统测试，编码对单元测试）。它强调在开发生命周期中早期进行验证和确认。
     *   **补充**：一旦编码完成，测试就从单元测试开始，层级逐步提高直到验收测试完成。它将测试与开发同等重视，而不是在最后将其视为事后的想法。
12.  **W Model (W模型 / 双V模型)**
     *   **解释**：V模型的扩展，强调测试活动与开发活动并行进行，而不仅仅是在编码完成后才开始。它体现了开发和测试之间的协作，测试贯穿整个生命周期。
     *   **补充**：它强调开发和测试的协作；测试不只是构建，还包括执行和评估。

## **三、 白盒测试技术**
16. **Control Flow Testing (控制流测试)**
    *   **解释**：一种结构测试策略，以程序的控制流为模型来设计测试。要求测试人员理解程序的逻辑结构，最适用于新软件的单元测试。
12. **Statement/Point Coverage (语句/点覆盖)**
    *   **解释**：设计测试用例，确保源代码中的每一条语句（每个点）至少被执行一次。这是最弱的逻辑覆盖标准。
13. **Decision/Branch/Edge Coverage (决策/分支/边覆盖)**
    *   **解释**：设计测试用例，确保程序中每个决策（分支）的真假分支至少被执行一次。满足分支覆盖一定满足语句覆盖，反之不成立。
14. **Condition Coverage (条件覆盖)**
    *   **解释**：对于每个包含多个布尔条件的决策，设计测试用例，确保每个条件都至少取一次真值和假值。**它不保证决策本身的真假结果都被覆盖。**
15. **Decision Condition Coverage (决策条件覆盖)**
    *   **解释**：条件覆盖和分支覆盖的组合。设计测试用例，确保每个决策的真假结果，以及决策中每个条件的真假结果至少出现一次。
16. **Condition Combination Coverage (条件组合覆盖)**
    *   **解释**：设计测试用例，使得每个决策中所有条件的可能组合情况都被测试。这是最强但开销最大的逻辑覆盖方法。使用真值表来识别所有可能的组合。
17. **Control Flow Graphs (控制流图)**
    - 解释：程序流图的简化版本，只描述程序的控制流（即计算机执行一个程序中语句的顺序）。其中每个节点表示一个或多个语句，每一条边表示一个“跳跃”或“分支”，有两个离开的边表示一个决策。请注意：它不展示对数据的特定操作以及分支或循环的特定条件。
18. **Path Coverage (路径覆盖)**
    *   **解释**：设计测试用例，确保程序中所有可能的执行路径（从入口到出口）至少被执行一次。路径覆盖能达到100%的语句覆盖和分支覆盖，但可能导致测试用例数量爆炸，尤其对于包含循环的程序。
19. **Basis Path Testing (基础路径测试)**
    - 解释：一种白盒测试方法，根据程序中可采取的流程或逻辑路径来定义测试用例。其目标是定义独立路径的数量，从而用最少的测试用例实现最大的路径覆盖率。
20. **Independent Path (独立路径)**
    - 解释：从程序入口到出口的路径，该路径至少有一条边（边缘）在任何其他路径中以前没有被遍历过。
21. **Cyclomatic Complexity (圈复杂度)**
    *   **解释**：衡量程序控制流复杂性的指标，计算公式有：V(G) = E - N + 2（边数-节点数+2）或 V(G) = P + 1（决策节点数+1）。它代表了程序中线性独立路径的最大数量，是基础路径测试中确定所需测试用例数目的依据。

## **四、 黑盒测试技术**
27. **Equivalence Partitioning (等价类划分)**
    *   **解释**：将输入域划分为若干个等价类，假设同一等价类中的值对程序的处理是等效的。从每个等价类中选取一个代表值作为测试输入，以减少测试用例数量。分为有效等价类（符合规则）和无效等价类（不符合规则）。
20. **Valid Equivalence Class & Invalid Equivalence Class (有效等价类与无效等价类)**
    - **解释**：
    - **Valid Equivalence Class (有效等价类)**：输入域中有意义的一组数据，用于**验证系统功能和性能是否能被准确地实施**。
      
    - **Invalid Equivalence Class (无效等价类)**：输入域中无意义的一组数据，用于**测试系统的容错性**。
21. **Boundary Value Analysis (BVA, 边界值分析)**
    *   **解释**：选取等价类边界及边界附近的值作为测试输入，因为许多错误发生在边界条件。边界值通常是最小值、最大值、刚好在范围内外的值等。
22. **Combinational Testing (组合测试)**

    - **解释**：识别软件**输入（原因）**的所有不同组合及其相关的输出效果的测试技术。因果关系被描述为基于软件规范的逻辑语句，常见的识别组合技术包括因果图、决策表和真值表。
23. **Truth Table (真值表)**

    - **解释**：用于测试程序所有不同行为而创建的表格，将输入（“原因”）和输出（“影响”）指定为布尔表达式。对于N个独立的原因，真值表将包含2^N种不同的组合，并指定软件在每种组合下的行为方式。
24. **Don’t Care Conditions (“不在乎”条件)**

    - **解释**：在真值表中，原因的值对结果没有影响的条件。这些条件用于减少规则的数列，因为在这些情况下，无论原因为真还是假，都会生成相同的输出，通常在真值表中用“*”表示。

25. **Decision Tables (决策表)**

    - **解释**：一种精确而紧凑地建模复杂逻辑的方式。它将条件与要执行的动作相关联，能够优雅地将**多个独立条件与多个动作关联起来，特别适用于有大量决策制定、输入变量间有重要逻辑关系或计算逻辑复杂的程序**。

26. **Random Testing (随机测试)**
    - **解释**：每个测试用例由一组（随机）输入值表示的测试方法。如果测试是完全自动化的，则每个测试用例由特定参数的值分布表示，通常包括上限和下限，以及用于在这些限制之间选择随机值的分布。
    
27. **Scenario Testing (场景测试)**
    *   **解释**：基于用例场景来设计测试，模拟用户**完成正常功能的核心业务逻辑操作以及主要的错误操作**，以验证软件的功能正确性和异常处理能力。一个场景通常包含一个基本流和若干备选流。
28. **Error Guessing (错误推测)**
    *   **解释**：基于测试人员的经验和直觉，**猜测程序中可能出错的地方，并设计相应的测试用例**（如空值、0、负数、超长字符串等）。它是其他测试技术的有效补充。

## **五、 测试级别与流程**
37. **Unit Testing (单元测试)**
    *   **解释**：针对软件中最小的可测试单元（如函数、方法）进行的测试，通常是白盒测试，由开发者完成。目的在于**尽早发现错误、帮助定位错误、提高代码质量**。
24. **FIRST Principle (FIRST原则)**
    - 解释：单元测试的关键原则，包含五个方面：Fast（快速，测试应秒级完成）、Isolated（隔离，用例间无依赖）、Repeatable（可重复，结果稳定不随机）、Self-Validating（自验证，自动判断通过/失败）、Timely（及时，测试与代码同步编写）。
25. **Integration Testing (集成测试)**
    *   **解释**：在单元测试之后，将已测试的模块组装起来进行测试，主要检查**模块之间的接口和交互是否正确**。常用方法有自顶向下（需桩模块）、自底向上（需驱动模块）、大突击和三明治集成。
26. **System Testing (系统测试)**
    *   **解释**：将经过集成测试的软件作为整个系统的一部分，与硬件、外部设备、操作系统等结合，在实际运行环境中进行测试。主要是黑盒测试，验证系统是否符合软件需求规格。
27. **Acceptance Testing (验收测试)**
    *   **解释**：软件测试的最终阶段，由用户或客户进行，以确认系统**是否符合业务需求及发行标准**。包括Alpha测试（开发者内部环境）和Beta测试（真实用户环境）。
28. **Drivers & Stubs (驱动模块和桩模块)**
    *   **解释**：
        *   **Driver (驱动模块)**：用于模拟被测模块的上层调用模块，**调用被测模块并传递输入，接收输出**。
        *   **Stub (桩模块)**：用于模拟被测模块调用的下层模块，为被测模块提供固定或有限的返回值。
29. **Regression Testing (回归测试)**
    *   **解释**：在软件修改（修复缺陷或增加新功能）后，重新执行之前通过的测试，以**确保修改没有引入新的错误或导致其他部分功能失效**。

## **六、 测试类型**

44. **Functional Testing (功能测试)**
    *   **解释**：验证软件的功能需求是否符合预期，包括正常输入和错误输入的处理。属于系统测试的一部分，主要使用黑盒测试技术。
30. **Performance Testing (性能测试)**
    *   **解释**：测试软件在特定负载下的响应速度、稳定性、资源使用情况等，**以识别和消除性能瓶颈**。包括负载测试、压力测试、耐久测试、尖峰测试等子类型。
31. **Load Testing (负载测试)**
    *   **解释**：测试软件在**预期用户负载下的性能表现**，**目标是找到性能瓶颈**。
32. **Stress Testing (压力测试)**
    *   **解释**：测试软件在**极端（超出正常）负载下的行为**，目标是**找到系统的崩溃点并观察其恢复能力**。
33. **Volume Testing (容量测试)**
    
    *   **解释**：测试软件在处理大量数据（如大量数据库记录）时的性能表现，目标是在**不同数据量下检查系统行为**。
    
34. **Baseline Test (基线测试)**

    - 解释：通过1个虚拟用户执行多轮次场景，验证应用程序性能是否符合业务服务级别协议（SLA）。其核心作用是**建立性能基准**，用于后续压力测试、负载测试的对比参照。

35. **Acceptance Testing (验收测试)**
    
    - 解释：软件测试的最终阶段，检验系统是否符合其业务要求。通常由客户或终端用户实施，在系统测试之后、最终发布之前进行，**使用黑盒测试方法**。**目的是检验产品是否符合发行标准**。
    
36. **Alpha Testing vs. Beta Testing (Alpha测试 vs. Beta测试)**

    - 解释：两者都是验收测试的类型和客户确认方法。

      - **Alpha Testing (Alpha测试)**：客户确认的第一步，在**开发者视角和受控的测试环境中进行**，不对外开放。发现的问题直接打日志并由开发者高优先级修复，通常执行多个测试周期。

      - **Beta Testing (Beta测试)**：客户确认的第二步，在**真实市场环境中由终端用户进行**，对外开放且不受开发者控制。问题以反馈形式提出，作为未来版本的优化参考，通常只执行1-2个测试周期，参与者常有奖励。

37. **Four Stages of Software Testing Comparison (软件测试四大阶段对比)**
    
    - 解释：V模型四大测试在多个维度上存在显著区别：
    
      - **测试对象**：单元测试针对软件单元；集成测试针对模块间接口；系统测试和验收测试针对整个系统（含软硬件）。
    
      - **测试基础**：单元测试依据详细设计；集成测试依据架构设计；系统测试依据需求确认；验收测试依据需求确认及验收协议。
    
      - **测试人员**：单元测试由开发/白盒测试人员完成；集成测试由开发+测试人员完成；系统测试由专业测试人员完成；验收测试由用户主导，开发和测试配合。
    
      - **测试方法**：单元测试以白盒为主；集成测试黑白结合；系统测试和验收测试以黑盒为主。
    
      - **测试数据**：单元和集成测试一般不使用真实数据；系统和验收测试尽可能使用或模拟真实业务数据。

## **七、 测试文档与维护**
53. **Version Control (版本控制)**

    *   **解释**：对软件系统的不同版本进行管理、追踪和协调的流程，是软件配置管理的重要组成部分，确保修改的可追溯性和一致性。

35. **Test Plan (测试计划)**

    - **解释**：一个描述软件测试范围以及行为的文档，是项目中正式测试任意软件/产品的基础。它定义了测试活动的范围、方法、资源和日程。包含Master Test Plan（针对项目/产品的高层级计划）、Testing Level Specific（针对测试层级的计划）和Testing Type Specific（针对特定测试类型的计划）。

36. **Test Case (测试用例)**

    - **解释**：由文档整理出来的一系列前置条件（前置需求）、流程（输入/行为）和后置条件（预期输出）。测试人员用其来决定系统是否符合需求或是否正确运行。一个测试用例可以有一个或多个测试脚本（用于执行测试的指令），测试用例的集合被称为测试包。

37. **Bug Report (缺陷报告)**
    *   **解释**：记录软件缺陷的文档，包括**缺陷描述、复现步骤、预期结果、实际结果、严重性和优先级等，目的是清晰传达问题以促进修复**。

38. **Test Summary Report (测试总结报告)**

    - 解释：在测试项目结束时或测试完成后准备的重要可交付成果。其主要目的是向项目干系人解释项目中关于被执行测试的各种细节及活动，包括发布测试结果、指标、最佳实践、经验教训和“上线”结论等，作为测试执行和测试结论的证据。

39. **Static Testing (静态测试)**

    - 解释：不执行代码，而是仔细且有条不紊地回顾和分析软件以发现BUG的过程。这种测试非常有价值，因为大量的错误可以通过静态测试被发现；从成本和生产力角度来看，它能在早期发现错误，相比基于执行的测试更节省时间和成本。

40. **Code Review (代码评审)**

    - 解释：静态测试的一种重要方法，包含四个基本元素：发现问题（发现软件忽略的条目、错误等）、遵守规则（规定被评审的代码量及花费时间等）、准备（参与者提前做准备）和书写报告（总结审查结果向开发团队提供报告）。常见形式包括非正式的同行评审和更正式的演练。

41. **Software Maintenance (软件维护)**
    *   **解释**：在软件交付后，为了纠正错误、提高性能或适应变化的环境而修改软件的过程。主要包括以下类型：
        *   **Corrective Maintenance (改正性维护)**：修复已发现的错误。
        *   **Adaptive Maintenance (适应性维护)**：使软件适应外部环境的变化（如操作系统、硬件）。
        *   **Perfective Maintenance (完善性维护)**：根据用户需求，增强软件功能或改进性能。
        *   **Preventive Maintenance (预防性维护)**：为提高未来的可维护性而进行的工作，如更新文档、重构代码。

42. **Software Configuration Management (SCM, 软件配置管理)**
    *   **解释**：在软件开发过程中，对软件配置项（如代码、文档、工具）的变更有序管理的过程。其流程通常包括：识别配置项 → 基线管理 → 版本控制 → 变更控制 → 配置审计 → 配置状态报告。

43. **Software Evolution (软件演变)**

    - 解释：将软件不断变好的持续改变过程。

44. **Software Maintainability (软件可维护性)**

    - 解释：软件可被修改的简单程度（难易程度）。

45. **Ripple Effect (涟漪效应)**

    - 解释：软件中一个位置的变化会影响其他组件的现象。

46. **Impact Analysis (影响分析)**

    - 解释：根据变化将如何影响系统的其余部分来确定变化的过程。

47. **Traceability (可追溯性)**

    - 解释：两个或多个软件工件（artifacts）之间可以建立关系的程度。

48. **Legacy Systems (遗留系统)**

    - 解释：仍在使用，但开发团队不再活跃的软件系统。

49. **Types of Software Maintenance (CAPP - 软件维护的类型)**

    - 解释：软件维护包含四种主要类型：

      - **Corrective Maintenance (改正性维护)**：修复潜在的错误（包括临时的补丁和变通方法 patches and workarounds）。

      - **Adaptive Maintenance (适应性维护)**：处理外部的变化（包括硬件平台的改变以及支持软件的改变）。

      - **Perfective Maintenance (完善性维护)**：提升已经发行的软件的用户体验以及效率。

      - **Preventive Maintenance (预防性维护)**：提升未来的可维护性（如改进文档、注释等）。
