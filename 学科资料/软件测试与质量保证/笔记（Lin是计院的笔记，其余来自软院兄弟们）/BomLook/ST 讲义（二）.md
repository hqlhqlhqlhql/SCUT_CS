# 软件测试与维护：讲义（二）黑盒测试去冗余笔记

## 一、整体脉络分析线

本讲义围绕 **黑盒测试如何从规格说明推导测试用例** 展开。

可以按照以下主线整理：

**黑盒测试概念 → 等价类划分 → 边界值分析 → 组合测试 → 决策表 → 随机测试与错误推测 → 场景测试 → 黑盒测试方法选择**

其中期末重点是：

1. **黑盒测试基本概念**：只基于规格说明，不依赖程序内部实现。
2. **等价类划分**：把输入域划分为有效等价类和无效等价类，从每类中选代表值。
3. **边界值分析**：在等价类边界附近选值，因为错误常发生在边界。
4. **组合测试 / 真值表 / 决策表**：处理多个输入条件组合导致不同输出的情况。
5. **场景测试**：针对业务流程，用基本流和备选流设计测试用例。

整体记忆线：

> 黑盒测试不是看代码，而是看规格；不是随便测，而是从规格中系统推导测试用例。

---

# 二、黑盒测试概念

## 1. 定义

**黑盒测试**完全基于程序规格说明，用于验证程序是否满足规定需求。

特点：

- 只关注输入和输出；
- 不关心程序内部结构；
- 能覆盖规格说明；
- 不能保证覆盖实现代码。

重要理解：

> 黑盒测试可以发现软件是否没有按规格工作，但不一定能发现实现中存在的额外行为。

## 2. 黑盒测试原则

黑盒测试应当：

1. 对照规格说明进行测试；
2. 使用基于规格说明的覆盖准则；
3. 从规格说明中推导测试用例；
4. 尽可能“执行”规格说明中规定的行为。

---

# 三、等价类划分 Equivalence Partitioning

## 1. 定义

**等价类划分**：将输入域划分为若干个等价类，同一等价类中的输入被认为会被程序以相同方式处理。

例如：

```java
boolean isNegative(int x) {
    return x < 0;
}
```

可划分为两个等价类：

- `Integer.MIN_VALUE <= x <= -1`：负数；
- `0 <= x <= Integer.MAX_VALUE`：非负数。

## 2. 什么是分区 Partition

等价类形成一个分区，分区满足：

- 各子集之间互不相交；
- 所有子集的并集覆盖整个输入域。

## 3. 等价类分类

| 类型 | 含义 | 作用 |
|---|---|---|
| 有效等价类 Valid Equivalence Class | 输入域中有意义、符合规格的一组数据 | 验证系统是否正确实现功能 |
| 无效等价类 Invalid Equivalence Class | 输入域中无意义、不符合规格的一组数据 | 测试系统容错能力 |

## 4. 识别等价类的步骤

### 步骤 1：识别输入 / 输出

根据规格说明找出：

- 输入参数；
- 输出结果；
- 输入约束；
- 输出约束。

### 步骤 2：根据输入 / 输出划分等价类

常见划分方式：

| 情况 | 划分方式 |
|---|---|
| 范围 | 有效范围 + 小于下界 + 大于上界 |
| 集合 | 有效集合 + 无效集合 |
| 布尔值 | true 类 + false 类 |
| 必填输入 | valid + null / empty 等无效类 |
| 规则约束 | 满足所有规则为有效类，违反任意规则为无效类 |

### 步骤 3：建立等价类表

为每个等价类编号，方便测试用例覆盖说明。

示例：百分制评分。

| 参数 | 有效等价类 | 无效等价类 |
|---|---|---|
| Score | `0 <= x <= 100`（1） | `x < 0`（2）；`x > 100`（3） |

### 步骤 4：设计测试用例

规则：

- 有效等价类：一个测试用例尽量覆盖多个有效类；
- 无效等价类：一个测试用例通常只覆盖一个无效类，其余输入保持有效。

这样做是为了避免多个无效输入同时出现时，无法判断到底是哪一个无效输入触发了错误处理。

# 四、弱等价类与强等价类

## 1. 弱等价类测试

弱等价类测试只要求每个等价类至少被覆盖一次。

测试用例数量通常取决于：

> 各输入参数中，等价类数量最多的那个参数。

适用于基本覆盖，测试用例数量少。

## 2. 强等价类测试

强等价类测试要求覆盖多个输入参数等价类之间的所有组合。

测试用例数量通常为：

> 各参数等价类数量的笛卡尔积。

例如：

- A 有 3 类；
- B 有 4 类；
- C 有 2 类；

则强等价类测试需要：

`3 × 4 × 2 = 24` 个测试用例。

## 3. 对比

| 类型 | 覆盖要求 | 用例数量 | 特点 |
|---|---|---|---|
| 弱等价类 | 每个等价类至少出现一次 | 少 | 成本低，组合覆盖弱 |
| 强等价类 | 覆盖所有等价类组合 | 多 | 覆盖强，成本高 |

# 五、等价类划分典型例题

## 例 1：外线电话号码

规格：电话号码必须是以 `9` 开头的 9 位数字串。

| 参数 | 有效等价类 | 无效等价类 |
|---|---|---|
| Phone Number | 以 `9` 开头的 9 位数字串（1） | 不以 `9` 开头（2）；包含非数字字符（3）；长度不是 9（4） |

测试用例：

| ID | 输入 | 预期输出 | 覆盖类 |
|---|---|---|---|
| 1 | `912345678` | OK | 1 |
| 2 | `874563210` | ERROR | 2 |
| 3 | `9qwerasdf` | ERROR | 3 |
| 4 | `98765` | ERROR | 4 |

## 例 2：Windows XP 文件名

规格：文件名不包含特殊字符，且长度在 1 到 256 之间。

| 参数 | 有效等价类 | 无效等价类 |
|---|---|---|
| File Name | 不含特殊字符且长度合法（1） | 包含特殊字符（2）；长度为 0（3）；长度超过 256（4） |

## 例 3：三角形类型判定

输入三条边 A、B、C。

### 一般三角形条件

有效等价类：

- `A > 0`；
- `B > 0`；
- `C > 0`；
- `A + B > C`；
- `A + C > B`；
- `B + C > A`。

无效等价类：

- `A <= 0`；
- `B <= 0`；
- `C <= 0`；
- `A + B <= C`；
- `A + C <= B`；
- `B + C <= A`。

### 等腰三角形条件

有效等价类：

- `A == B`；
- `B == C`；
- `A == C`。

无效等价类：

- 三边互不相等。

### 等边三角形条件

有效等价类：

- `A == B && B == C && A == C`。

无效等价类：

- `A != B`；
- `B != C`；
- `A != C`。

考试注意：

> 三角形问题通常不是只测“能不能构成三角形”，还要继续判断一般三角形、等腰三角形、等边三角形。

## ==例 4：seatsAvailable()==

输入参数：

- 空余座位数；
- 需求座位数。

核心等价类：

| 参数 | 等价类 |
|---|---|
| 空余座位 | 小于 0；不足需求；满足需求；超过座位总数 |
| 需求座位 | 小于等于 0；小于等于空余座位；大于空余座位但不超过总座位；超过总座位 |

有效类测试应尽量组合覆盖；无效类测试应一次只覆盖一个无效输入。

# 六、等价类划分总结

## 1. 优点

- 提供基本测试覆盖；
- 适合输入变量清晰、取值范围明确的数据处理类应用；
- 能结构化地产生基本测试用例；
- 用例数量较少，成本低。

## 2. 缺点

- 不关注分区边界；
- 不充分测试输入组合；
- 没有自动算法保证能找到所有合理分区；
- 对规格说明质量依赖较高。

## 3. 结论

> 等价类划分是最低限度的黑盒测试方法，适合建立基础测试集，但还需要结合边界值分析和组合测试。

---

# 七、边界值分析 Boundary Value Analysis, BVA

## 1. 定义

**边界值分析**关注输入或输出范围边缘的测试数据。

原因：

> 软件错误经常发生在边界附近。

例如：

- 数组越界；
- 缓冲区溢出；
- 循环条件错误；
- 最大 / 最小长度处理错误。

## 2. 边界值选取原则

每个参数在每个等价类的上界和下界附近都应考虑测试。

常见边界：

| 类型 | 典型边界 |
|---|---|
| 数值 | 最小值、最大值、刚小于边界、刚大于边界 |
| 循环 | 0、1、`len - 1`、`len` |
| 数组 | 第一个元素、最后一个元素 |
| 字符串 | 空串、最短合法长度、最长合法长度、超过最长长度 |
| 链表 | 第一个节点、最后一个节点、空链表、单节点链表 |
| 表格 | 第一行、最后一行、第一列、最后一列 |

## 3. 示例：isNegative

```java
boolean isNegative(int x) {
    return x < 0;
}
```

等价类：

- `[Integer.MIN_VALUE, -1]`；
- `[0, Integer.MAX_VALUE]`。

边界值可选：

- `Integer.MIN_VALUE`；
- `-1`；
- `0`；
- `Integer.MAX_VALUE`。

## 4. 优缺点

| 优点 | 缺点 |
|---|---|
| 测试数据选择明确 | 不测试输入组合 |
| 聚焦更容易出错的区域 | 需要结合其他方法 |

结论：

> 边界值分析通常应和等价类划分一起使用。

# 八、组合测试 Combinational Testing

## 1. 定义

组合测试关注多个输入条件之间的组合，以及这些组合导致的输出结果。

常用方法包括：

- 真值表；
- 决策表；
- 因果图。

适用场景：

- 多个输入条件共同影响输出；
- 条件之间存在逻辑关系；
- 业务规则复杂；
- 程序中有大量条件判断。

## 2. 真值表 Truth Table

真值表将输入条件和输出结果表示为布尔逻辑。

对于 N 个独立原因，最坏情况下有：

`2^N` 种组合。

示例：`isNegative(int x)`

| Cause | Rule 1 | Rule 2 |
|---|---|---|
| `x < 0` | T | F |
| return true | T | F |
| return false | F | T |

## 3. Don't Care 条件

当某个条件取值不影响输出时，可以用 `*` 表示。

作用：

- 减少规则数量；
- 合并等价规则；
- 降低测试用例数量。

示例：

```java
return x < 0 && flag ? true : false;
```

| Cause | Rule 1 | Rule 2 | Rule 3 |
|---|---|---|---|
| `x < 0` | T | F | * |
| `flag` | T | T | F |
| return | T | F | F |

解释：

- 当 `flag = false` 时，无论 `x < 0` 是否成立，返回值都是 false；
- 因此 `x < 0` 是 Don't Care。

# 九、决策表 Decision Table

## 1. 定义

决策表是一种精确且紧凑的方式，用来表达复杂业务逻辑中：

- 条件 Conditions；
- 动作 Actions；
- 条件组合与动作之间的对应关系。

决策表比真值表更适合实际业务规则。

## 2. 适用场景

当规格说明具有以下特点时，适合使用决策表：

- 有多个输入条件；
- 输入条件之间有重要逻辑关系；
- 不同条件组合触发不同动作；
- 业务逻辑复杂；
- 圈复杂度较高。

## 3. NextDate Program 决策表思想

NextDate 问题输入：

- 年 Year；
- 月 Month；
- 日 Day。

输出：

- 下一天日期；
- 或 Invalid。

初始划分：

- 月份：30 天月份、31 天月份、2 月；
- 日期：1..28、29、30、31；
- 年份：闰年、平年。

如果直接做真值表，组合数量会很大。优化思路是：

1. 利用月份类别互斥性减少组合；
2. 将 12 月从 31 天月份中单独拆出，因为它涉及跨年；
3. 将 2 月单独处理，因为它涉及闰年和平年；
4. 对不影响输出的条件进行合并。

优化后的关键分类：

| 分类 | 内容 |
|---|---|
| M1 | 30 天月份 |
| M2 | 31 天月份，不含 12 月 |
| M3 | 12 月 |
| M4 | 2 月 |
| D1 | 1..27 |
| D2 | 28 |
| D3 | 29 |
| D4 | 30 |
| D5 | 31 |
| Y1 | 闰年 |
| Y2 | 平年 |

注意：

- 除 2 月外，其他月份通常不需要考虑闰年 / 平年；
- 12 月 31 日需要考虑跨年；
- 2 月 28 / 29 日需要结合闰年判断。

## 4. NextDate 常见测试点

| 情况 | 示例 | 预期输出 |
|---|---|---|
| 普通日期 | 2001-04-15 | 2001-04-16 |
| 30 天月份月末 | 2001-04-30 | 2001-05-01 |
| 30 天月份非法 31 日 | 2001-04-31 | Invalid |
| 31 天月份普通日 | 2001-01-15 | 2001-01-16 |
| 31 天月份月末 | 2001-01-31 | 2001-02-01 |
| 年末 | 2001-12-31 | 2002-01-01 |
| 闰年 2 月 28 日 | 2004-02-28 | 2004-02-29 |
| 平年 2 月 28 日 | 2001-02-28 | 2001-03-01 |
| 闰年 2 月 29 日 | 2004-02-29 | 2004-03-01 |
| 平年 2 月 29 日 | 2001-02-29 | Invalid |
| 2 月 30 日 | 2001-02-30 | Invalid |

说明：原资料中个别 NextDate 表格行存在明显可疑处，例如“平年 2 月 28 日输出 Invalid”不符合题目定义；按题目前置代码和常规 NextDate 规则，应为下一天 `2001-03-01`。

## 5. 超市会员决策表示例

输入条件：

- 是否会员；
- 购买金额是否大于等于 1000；
- 是否现场办理会员卡。

输出动作：

- 七折；
- 九折；
- 原价。

设计思想：

- 不同条件组合对应不同折扣；
- 不影响结果的条件可用 `-` 或 `*` 合并；
- 最终为每一列规则设计至少一个测试用例。

# 十、随机测试与错误推测

## 1. 随机测试 Random Testing

随机测试从输入域中随机选取测试数据。

特点：

- 可用于自动化测试；
- 可设置输入分布和上下限；
- 适合作为系统性测试方法的补充。

## 2. 错误推测 Error Guessing

错误推测依赖测试人员经验，选择更可能暴露缺陷的数据。

常见错误推测数据：

- null；
- 空字符串；
- 空数组；
- 空列表；
- 空对象引用；
- 0；
- 负数；
- 含空格字符串；
- 已排序数据；
- 逆序数据；
- 重复元素；
- 单元素集合。

示例：排序算法可测试：

- `[]`；
- `[1]`；
- `[1, 2, 3, 4, 5]`；
- `[9, 7, 5, 4, 3]`；
- `[3, 4, 4, 3, 9]`。

## 3. 特点

优点：

- 对有经验测试人员很有效；
- 能补充系统化方法遗漏的特殊情况。

缺点：

- 依赖经验；
- 不够系统；
- 不能单独作为主要测试方法。

# 十一、场景测试 Scenario Testing

## 1. 定义

场景测试根据用户业务流程设计测试用例，模拟用户完成正常功能或触发异常流程。

目标：

1. 模拟用户完成正常业务流程，验证核心功能正确性；
2. 模拟用户常见错误操作，验证异常处理能力。

## 2. 基本流与备选流

| 概念 | 含义 |
|---|---|
| 基本流 Basic Flow | 用户完成主要功能的正常流程 |
| 备选流 Alternative Flow | 异常、分支、错误处理或可选路径 |

注意：

- 备选流太多会导致场景爆炸；
- 应选择典型场景；
- 每个备选流至少被一个场景覆盖；
- 尽量避免一个异常测试用例同时覆盖多个备选流。

## 3. 场景测试设计步骤

1. 根据规格说明描述基本流和备选流；
2. 构造满足覆盖要求且无冗余的场景；
3. 为每个场景设计测试用例；
4. 检查并删除冗余测试用例；
5. 为每个测试用例确定具体测试数据。

## 4. 酒店预订系统示例

业务流程：

1. 用户选择预订日期；
2. 选择房间；
3. 登录账户；
4. 输入密码；
5. 支付押金；
6. 系统生成预订表单。

约束：

- 可预订期限为 30 天；
- 押金为 400 美元。

测试场景：

| 测试用例 | 场景 | 日期 | 房间 | 账户 | 密码 | 余额 | 预期输出 |
|---|---|---|---|---|---|---|---|
| 1 | 正常预订 | 有效 | 未满 | 存在 | 正确 | 充足 | 成功预订，扣除押金 |
| 2 | 日期无效 | 无效 | NA | NA | NA | NA | 提示重新选择日期 |
| 3 | 房间已满 | 有效 | 已满 | NA | NA | NA | 提示房间已满 |
| 4 | 账户不存在 | 有效 | 未满 | 不存在 | NA | NA | 提示账户不存在 |
| 5 | 密码错误 | 有效 | 未满 | 存在 | 错误 | NA | 提示密码错误 |
| 6 | 余额不足 | 有效 | 未满 | 存在 | 正确 | 不足 | 提示余额不足 |

说明：

- `V` 表示有效；
- `I` 表示无效；
- `NA` 表示该条件在当前场景中不适用。

# 十二、黑盒测试方法选择总结

| 情况 | 推荐方法 |
|---|---|
| 输入字段明确、取值范围清楚 | 等价类划分 |
| 输入存在上下界或长度限制 | 边界值分析 |
| 多个条件组合影响输出 | 决策表 / 真值表 |
| 业务逻辑复杂 | 决策表 |
| 业务流程清晰 | 场景测试 |
| 需要补充特殊情况 | 错误推测 |
| 自动化随机探索 | 随机测试 |

综合策略：

1. 先用等价类划分得到基本测试集；
2. 再用边界值分析补充边界数据；
3. 若存在复杂条件组合，使用决策表；
4. 若存在完整业务流程，使用场景测试；
5. 最后用错误推测补充容易遗漏的特殊输入。

# 十四、最终压缩版记忆框架

可以用下面这条线复习：

**黑盒测试看什么？**  
看规格说明，不看代码。

**怎么从规格说明生成测试？**  
先找输入 / 输出，再划分等价类。

**怎么减少测试数量？**  
有效类尽量合并覆盖，无效类单独测试。

**哪里最容易出错？**  
边界，所以要做边界值分析。

**多个条件共同决定结果怎么办？**  
用真值表或决策表。

**业务流程怎么测？**  
用场景测试：基本流 + 备选流。

**系统方法之外还缺什么？**  
用错误推测补充特殊情况。
