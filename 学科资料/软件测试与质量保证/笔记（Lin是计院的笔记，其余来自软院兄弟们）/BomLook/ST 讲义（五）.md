# 三、Ch6 测试文档

## 1. Test Plan 测试计划

定义：测试计划是描述测试范围、方法、资源和进度的文档，是正式开展测试活动的基础。

类型：

- 主测试计划（Master Test Plan）：项目/产品层面的高层级统一计划。
- 测试级别特定计划：单元测试、集成测试、系统测试、验收测试计划。
- 测试类型特定计划：性能测试、安全测试等专项测试计划。

流程：定义测试策略 → 定义测试系统 → 预估测试成本 → 准备并回顾测试计划。

## 2. Test Case 测试用例

定义：测试用例是用于判断系统是否符合需求或正确运行的一组条件与步骤，通常包括前置条件、输入/操作流程、预期输出与后置条件。

- 一个测试用例可对应一个或多个测试脚本（test scripts）。
- 测试脚本是一系列执行测试的指令。
- 测试用例集合称为测试包（test suite）。

## 3. Bug Report 缺陷报告

定义：缺陷报告用于说明软件出现了什么问题、为什么被视为错误，以及期望如何修复。

目标：推动缺陷被修复，即说明“当前结果是什么、期望结果是什么、为什么需要修复”。

要求：清晰、可执行、易于复现或处理。

## 4. Test Summary Report 测试总结报告

定义：测试总结报告是在测试完成后形成的重要交付物，用于向干系人说明测试活动、测试结果和测试结论。

核心内容：测试结果、度量指标、最佳实践、经验教训、上线/不予上线结论，以及作为测试已执行的证据。

# 四、Ch10 静态测试

考试范围：2025 PPT 1–15；Static Program Analysis 不考。

## 1. 静态测试定义与价值

定义：静态测试是在不执行代码的情况下，通过有组织的评审和分析发现软件缺陷的过程。

- 价值：能在早期发现大量缺陷。
- 成本优势：越早发现并修复缺陷，时间和成本越低。
- 与动态测试区别：静态测试看文档/代码本身，动态测试运行程序观察行为。

## 2. Code Review 代码评审

代码评审的四个基本元素：

- 发现问题：识别遗漏、错误与潜在缺陷。
- 遵守规则：明确评审范围、代码量、时间与标准。
- 准备：参与者在评审前阅读并准备意见。
- 书写报告：记录评审结果并反馈给开发团队。

非正式代码评审：

- 同行评审：小规模程序员/测试人员作为审阅者，仍应遵循四个基本元素。
- 演练（Walkthrough）：作者向审阅者逐行展示并解释代码，审阅者提问。

## 3. 代码评审检查表

- **数据引用错误：**检查跨函数引用的数据结构是否定义一致；注意 off-by-one 等边界错误。
- **数据声明错误：**检查变量是否在模块内显式声明。
- **计算错误：**检查算术、类型转换、精度、溢出等问题。
- **比较错误：**检查条件判断、边界条件、相等/不等比较是否正确。
- **控制流错误：**检查循环、分支、异常路径是否完整且可达。
- **子程序参数错误：**检查实参与形参类型、顺序、数量和含义是否一致。
- **输入输出错误：**检查外部输入/输出格式是否符合规范，例如日期格式。
- **其他错误项：**检查注释、命名、资源释放、可读性和可维护性等。

# 五、Ch12 软件维护

## 1. 软件维护定义

定义：软件维护是在软件交付后，对系统或组件进行修改，以纠正故障、提升性能/属性，或适应环境变化的过程。

- 维护通常不涉及系统架构的大规模改变。
- 维护需要程序理解。
- 软件维护强调部署后保持系统可操作；软件演化强调持续改进。

## 2. 必考：维护类型 CAPP

类型

中文

重点含义

Corrective

纠错性维护

修复已发现或潜在错误，包括补丁和变通方案。

Adaptive

适应性维护

适应外部环境变化，如硬件平台或支持软件变化。

Perfective

完善性维护

提升用户体验、效率或其他已有功能质量。

Preventative

预防性维护

提升未来可维护性，如改进文档、注释和结构。

## 3. 软件维护生命周期与主要活动

维护变更通过软件维护过程实施，主线如下：

识别变化（是什么、为什么） → 管理过程（资源与优先级） → 理解程序（如何改、影响范围） → 做出改变 → 测试改变 → 文档记录与变更记录。

## 4. 可维护性与维护术语

- **Maintainability 可维护性：**软件被修改的难易程度。
- **Ripple effect 涟漪效应：**某一位置的变更可能影响其他组件。
- **Impact analysis 影响分析：**识别某项变更会如何影响系统其他部分的过程。
- **Traceability 可追溯性：**两个或多个软件工件之间可建立关系的程度。
- **Legacy systems 遗留系统：**仍在使用，但原开发团队已不活跃的软件系统。
