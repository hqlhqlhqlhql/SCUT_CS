# 补充

## 可用性测试 Usability Testing

可用性测试属于**系统测试中的非功能测试**，主要验证系统是否**容易学习、容易使用、少出错、用户满意**。它关注的不是功能是否存在，而是用户能否高效、顺利、舒服地使用系统。

------

## 一、测试目标

可用性测试主要回答：

1. 用户能否快速学会使用系统；
2. 用户完成任务是否高效；
3. 用户是否容易犯错；
4. 用户过一段时间后是否还能记得如何使用；
5. 用户是否喜欢这个界面。

------

## 二、五个评价指标

| 指标       | 含义                                 |
| ---------- | ------------------------------------ |
| 学习时间   | 用户学会基本操作需要多久             |
| 使用速度   | 完成任务需要多少点击、输入或页面跳转 |
| 用户错误率 | 用户在操作中犯错的频率               |
| 技能保持   | 用户隔一段时间后是否还能记得操作方法 |
| 主观满意度 | 用户是否觉得界面好用、舒服、不烦人   |

------

## 三、常用测试方法

| 方法       | 特点                                          |
| ---------- | --------------------------------------------- |
| 问卷       | 易重复，适合收集用户偏好，建议至少 30 份      |
| 访谈       | 能深入了解原因，但耗时，建议约 5 人           |
| 观察       | 看用户真实操作过程，建议 3 人以上             |
| 出声思考   | 用户边操作边说想法，适合发现误解，建议 3–5 人 |
| 启发式评估 | 根据可用性准则检查界面问题                    |

------

## 四、启发式评估关注点

主要检查：

1. 界面是否简单自然；
2. 是否使用用户能理解的语言；
3. 是否减少用户记忆负担；
4. 操作是否一致；
5. 是否有及时反馈；
6. 是否有明确退出方式；
7. 是否提供快捷方式；
8. 错误信息是否清楚；
9. 是否能预防错误；
10. 帮助和文档是否易查找。

------

## 五、一句话总结

**可用性测试就是从用户角度检查系统是否“好学、好用、不容易出错、用起来舒服”。**





下面是**系统测试中：性能测试 Performance Testing 的精简笔记**。

## 一、性能测试是什么

性能测试，也叫 **Perf Testing**，是系统测试中的一种**非功能测试**，用于检查软件在一定工作负载下的表现，重点关注系统的：

| 关注点               | 含义                           |
| -------------------- | ------------------------------ |
| Speed 速度           | 系统响应是否足够快             |
| Scalability 可扩展性 | 系统最多能承受多少用户或数据量 |
| Stability 稳定性     | 系统在不同负载下是否稳定运行   |

性能测试的目标是发现并消除系统中的**性能瓶颈**，判断软件是否满足速度、扩展性和稳定性要求。

------

## 二、为什么要做性能测试

功能正确不代表系统可用。性能差可能导致用户体验差、系统崩溃、业务损失，甚至影响关键任务系统的安全运行。

常见原因包括：

1. 上线前发现性能问题；
2. 避免响应慢、崩溃、宕机；
3. 验证系统是否能承受预期用户量；
4. 找出 CPU、内存、网络、磁盘等瓶颈；
5. 保证关键系统长时间稳定运行。

------

## 三、常见性能问题

| 问题            | 含义                         |
| --------------- | ---------------------------- |
| 加载时间长      | 应用启动或页面加载太慢       |
| 响应时间差      | 用户输入后，系统返回结果太慢 |
| 可扩展性差      | 无法支持预期用户数量         |
| 瓶颈 Bottleneck | 某些资源限制导致整体性能下降 |

常见瓶颈来源：

- CPU 利用率过高；
- 内存占用过高；
- 网络带宽不足；
- 磁盘 I/O 过高。

------

## 四、性能测试常用指标

### 1. 客户端指标

| 指标                       | 含义                                   |
| -------------------------- | -------------------------------------- |
| Transaction Response Time  | 事务响应时间                           |
| Throughput                 | 吞吐量，即单位时间内服务器传输的数据量 |
| Hits/second                | 每秒 HTTP 请求数                       |
| Passed/Failed Transactions | 成功/失败事务数量                      |
| Transaction Error Rate     | 事务错误率                             |

### 2. 系统与网络指标

常监控：

- CPU 使用率；
- 内存使用率；
- 磁盘使用率；
- 网络延迟；
- 数据库响应时间；
- 数据库锁、连接池、SQL 执行情况等。

------

## 五、性能测试类型

| 类型                       | 目的                             |
| -------------------------- | -------------------------------- |
| Load Testing 负载测试      | 检查系统在预期用户负载下的表现   |
| Stress Testing 压力测试    | 在极端负载下测试系统极限和崩溃点 |
| Endurance Testing 耐久测试 | 检查系统长时间运行是否稳定       |
| Spike Testing 尖峰测试     | 检查系统面对突然流量激增的反应   |
| Volume Testing 容量测试    | 在大量数据条件下测试系统表现     |

简单区分：

- **负载测试**：看系统在正常或预期高负载下能不能正常工作；
- **压力测试**：故意超过系统承受范围，找崩溃点；
- **容量测试**：重点看大量数据对系统性能的影响。

------

## 六、性能测试、负载测试、压力测试区别

| 对比项   | 性能测试                           | 负载测试                   | 压力测试                   |
| -------- | ---------------------------------- | -------------------------- | -------------------------- |
| 范围     | 最大，是总称                       | 性能测试的子集             | 性能测试的子集             |
| 目标     | 建立性能基准                       | 找系统在预期负载下的上限   | 找系统极限和崩溃点         |
| 负载范围 | 可低于或高于崩溃阈值               | 通常到崩溃阈值前           | 超过崩溃阈值               |
| 关注点   | 响应时间、吞吐量、资源使用、稳定性 | 并发用户、响应时间、吞吐量 | 过载、恢复能力、内存泄漏等 |

------

## 七、性能测试流程

性能测试一般包括 7 步：

1. **识别测试环境**
   明确硬件、软件、网络配置，以及测试工具。
2. **确定性能验收标准**
   明确响应时间、吞吐量、资源使用率等目标。
3. **计划和设计测试**
   选择关键业务场景，设计用户负载和测试数据。
4. **配置测试环境**
   准备测试环境、工具和资源。
5. **实现测试设计**
   编写或录制性能测试脚本。
6. **运行测试**
   执行测试并监控系统表现。
7. **分析、调优、重新测试**
   分析结果，定位瓶颈，优化后再次测试。

------

## 八、常用性能测试工具

| 工具          | 特点                                 |
| ------------- | ------------------------------------ |
| HP LoadRunner | 可模拟大量虚拟用户，适合复杂性能测试 |
| Apache JMeter | 常用于 Web 和应用服务器的负载测试    |

LoadRunner 常见组成：

1. **VUGen**：录制和生成虚拟用户脚本；
2. **Controller**：设计和执行负载场景，例如控制虚拟用户数量、增长速度、并发方式；
3. **Analysis**：分析测试结果并生成图表，帮助定位瓶颈。

------

## 九、一句话总结

**性能测试就是在不同负载条件下，检查系统是否响应快、能承受足够用户、长时间稳定运行，并找出 CPU、内存、网络、数据库等性能瓶颈。**



下面是**系统测试中：安全测试 Security Testing 的精简笔记**。

## 一、安全测试是什么

安全测试属于系统测试中的**非功能测试**，用于发现系统中的安全漏洞，验证系统的数据和资源是否能防止入侵者攻击。其核心目标是保护系统资产，如服务器、应用、网页和数据，避免被破坏、未授权访问、拒绝服务或恶意软件攻击。

安全测试主要关注三类损失：

| 安全属性               | 含义                         |
| ---------------------- | ---------------------------- |
| Integrity 完整性       | 数据不能被非法篡改           |
| Confidentiality 机密性 | 数据不能被非法泄露           |
| Availability 可用性    | 系统不能被攻击到不可用或极慢 |

------

## 二、安全测试的目标

安全测试的主要目标包括：

1. 识别系统中的威胁；
2. 衡量系统潜在漏洞；
3. 发现可能的安全风险；
4. 帮助开发人员通过修改代码修复安全问题。

简单说：**安全测试就是找出系统哪里可能被攻击、攻击会造成什么后果，以及如何修复。**

------

## 三、核心概念

| 概念               | 含义                         |
| ------------------ | ---------------------------- |
| Threat 威胁        | 可能危害系统的事件           |
| Vulnerability 漏洞 | 攻击者可以利用的系统弱点     |
| Risk 风险          | 威胁利用漏洞后可能造成的损失 |

可理解为：

> **Threat + Vulnerability = Risk**
> 威胁利用漏洞，就会形成风险。

------

## 四、安全测试的三种主要方法

资料中把安全测试方法分为三类：**威胁建模、漏洞扫描、渗透测试**。

| 方法                            | 作用                               |
| ------------------------------- | ---------------------------------- |
| Threat Modeling 威胁建模        | 从设计阶段分析系统可能被怎样攻击   |
| Vulnerability Scanning 漏洞扫描 | 用自动化工具扫描已知漏洞           |
| Penetration Testing 渗透测试    | 模拟真实攻击，验证漏洞是否可被利用 |

------

## 五、威胁建模 Threat Modeling

威胁建模用于帮助团队分析系统的攻击面、评估威胁风险，并推动漏洞缓解措施。它主要回答三个问题：

1. 系统哪里最容易被攻击？
2. 最相关的威胁是什么？
3. 应该采取什么防护措施？

### 威胁建模流程

| 步骤              | 内容                               |
| ----------------- | ---------------------------------- |
| 1. 理解安全需求   | 明确使用场景、外部依赖、安全假设   |
| 2. 建立活动矩阵   | 分析角色、资产和操作关系           |
| 3. 创建信任边界   | 找出系统边界和潜在攻击路径         |
| 4. 识别威胁和攻击 | 可使用威胁树、滥用案例等方法       |
| 5. 评估风险并排序 | 判断哪些风险优先处理               |
| 6. 制定缓解措施   | 通过认证、授权、输入验证等方式防护 |

------

## 六、STRIDE 威胁模型

Microsoft STRIDE 是常用的威胁分类方法。

| STRIDE                 | 中文含义            | 破坏的安全属性             |
| ---------------------- | ------------------- | -------------------------- |
| Spoofing               | 电子欺骗 / 冒充身份 | Authentication 认证        |
| Tampering              | 篡改数据或代码      | Integrity 完整性           |
| Repudiation            | 否认行为            | Non-repudiation 不可否认性 |
| Information Disclosure | 信息泄露            | Confidentiality 机密性     |
| Denial of Service      | 拒绝服务            | Availability 可用性        |
| Elevation of Privilege | 权限提升            | Authorization 授权         |

------

## 七、常见防护措施

威胁建模后，需要制定缓解措施，常见措施包括：

1. **认证 Authentication**：用户名和密码验证，禁止默认账号和后门账号；
2. **授权 Authorization**：基于角色控制访问权限，采用最小权限原则；
3. **Cookie 管理**：保护会话和身份信息；
4. **输入验证 Data/Input Validation**：防止 SQL 注入、脚本注入等；
5. **错误处理 Error Handling**：避免错误信息泄露敏感数据；
6. **日志与审计 Logging/Auditing**：记录关键操作，便于追踪；
7. **加密 Cryptography**：保护传输和存储数据；
8. **会话管理 Session Management**：防止会话劫持和非法访问。

------

## 八、漏洞扫描 Vulnerability Scanning

漏洞扫描是发现、分析和报告安全缺陷的过程，通常通过自动化工具完成。它用于识别网络、硬件、软件和系统中的潜在风险和攻击路径。

常见漏洞扫描类型：

| 类型                                  | 说明             |
| ------------------------------------- | ---------------- |
| Port Scanner                          | 端口扫描         |
| Web Application Vulnerability Scanner | Web 应用漏洞扫描 |
| Network Vulnerability Scanner         | 网络漏洞扫描     |
| Host-based Vulnerability Scanner      | 主机漏洞扫描     |
| Database Scanner                      | 数据库漏洞扫描   |
| Source Code Vulnerability Scanner     | 源代码漏洞扫描   |
| Cloud Vulnerability Scanner           | 云环境漏洞扫描   |

常见工具包括 Nessus、Qualys、Acunetix、Metasploit、Nmap 等。

------

## 九、常见软件漏洞

资料中列举的常见漏洞包括：

| 漏洞                       | 含义                                               |
| -------------------------- | -------------------------------------------------- |
| Buffer Overflow 缓冲区溢出 | 读写超出缓冲区边界，可能导致系统崩溃或执行恶意代码 |
| SQL Injection SQL 注入     | 通过恶意 SQL 操作数据库                            |
| Weak Password 弱密码       | 密码容易被猜测或破解                               |
| HTTP Trace                 | 可能被利用来获取敏感请求信息                       |

其中，缓冲区溢出常见于 C/C++ 程序，攻击者可能通过覆盖返回地址等方式执行任意代码。

------

## 十、渗透测试 Penetration Testing

渗透测试，也叫 **Pen Test** 或 **Ethical Hacking**，是通过模拟真实攻击来评估系统安全风险的方法。它的目标是发现可被攻击者利用的漏洞，并评估攻击成功后的影响。

### 渗透测试分类

| 分类              | 含义                         |
| ----------------- | ---------------------------- |
| External 外部测试 | 从外部攻击者角度测试         |
| Internal 内部测试 | 从恶意员工或内部人员角度测试 |
| Overt 公开测试    | IT 部门知道测试正在进行      |
| Covert 隐蔽测试   | IT 部门不知道测试正在进行    |

### 渗透测试阶段

1. Reconnaissance and Information Gathering：侦察与信息收集；
2. Network Enumeration and Scanning：网络枚举与扫描；
3. Vulnerability Testing and Exploitation：漏洞测试与利用；
4. Reporting：报告结果。

------

## 十一、漏洞扫描 vs 渗透测试

| 对比项 | 漏洞扫描               | 渗透测试                         |
| ------ | ---------------------- | -------------------------------- |
| 方式   | 多为自动化工具扫描     | 人工主导，模拟真实攻击           |
| 目标   | 找出已知漏洞并报告风险 | 利用漏洞，验证攻击影响           |
| 深度   | 较浅，覆盖面广         | 较深，更接近真实攻击             |
| 关系   | 是渗透测试的前置基础   | 通常基于漏洞扫描结果继续攻击验证 |

------

## 十二、一句话总结

**安全测试就是通过威胁建模、漏洞扫描和渗透测试，发现系统可能被攻击的地方，验证数据和资源是否能保持机密性、完整性和可用性，并帮助开发人员修复安全问题。**
