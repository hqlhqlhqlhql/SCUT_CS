# 软件测试与维护：讲义（一）去冗余笔记

# 一、整体脉络分析线

本讲义的主线可以整理为：

**为什么需要测试 → 测试到底发现什么 → 如何判断软件是否正确 → 测试在开发流程中的位置 → 常见测试思想与方法**

具体展开为：

1. **软件与质量风险**  

   软件系统由程序、数据结构、配置文件、系统文档、用户文档等组成。现代软件规模大、需求不精确、开发周期和预算受限，因此质量风险高。

2. **测试的核心目标** 

   软件测试是用于**判断质量、发现错误**的系统方法。测试无法证明软件没有缺陷，只能尽可能早地发现缺陷并降低风险。

3. **缺陷产生与暴露机制**  

   重点理解 **Fault → Error → Failure**，以及 PIE 模型中的 **Execution → Infection → Propagation**。这是测试能否发现问题的核心逻辑。

4. **规格说明与正确性的判断**  

   软件是否有 Bug，必须依赖规格说明或预期行为判断。规格说明需要描述正常行为和异常行为。

5. **Verification 与 Validation**  

   Verification 关注“是否正确地构建产品”，偏过程；Validation 关注“构建的是否是正确产品”，偏结果。

6. **测试流程模型**  

   从瀑布模型、V 模型、W 模型，到敏捷、TDD、CI，测试逐渐从后置活动变成贯穿生命周期的持续活动。

7. **测试技术分类**  

   静态验证不执行代码，动态验证执行代码。黑盒测试依赖规格说明，白盒测试依赖代码结构和规格说明。

# Ch1 Introduction

## 1. 什么是软件？

软件系统通常包括：

- 程序指令；
- 数据结构；
- 配置文件；
- 系统文档；
- 用户文档。

早期软件的问题：

- 项目超支
- 项目超时
- 质量低下
- 无法满足需求
- 项目混乱
- 软件难以维护

现代软件的特点：

- 软件市场规模更大，竞争更激烈；
- 用户数量更多，使用场景更复杂；
- 嵌入式软件广泛存在于航空、航天、手表、烤炉、遥控器等设备中；
- 敏捷开发缩短交付周期，也增加了测试人员保障质量的压力。

## 2. 软件质量风险

软件开发存在以下风险：

- 程序规模大，复杂度高；
- 需求可能模糊、不精确；
- 截止日期和预算限制会压缩质量保障活动。

质量差可能导致：

- 软件失效；
- 高维护成本；
- 部署延迟；
- 声誉损失；
- 法律索赔；
- 市场份额下降。

国际标准可提供：  

- 支持设定质量目标。  

- 支持设计审查、验证和确认。 

国际标准：  

1. 质量模型围绕六个主要属性及其子特征构建。
2. 质量可以通过客观和主观指标的混合来衡量。
3. 适用于所有软件产品和计算机系统。

# 二、重点概念：Fault、Error、Failure

## 1. 三者定义

| 概念 | 中文 | 核心含义 | 位置 |
|---|---|---|---|
| Fault | 故障 / 缺陷 | 软件中的静态缺陷 | 代码、设计、文档中 |
| Error | 错误 | 由 Fault 引发的不正确内部状态 | 程序运行内部 |
| Failure | 失效 | 外部可观察到的不正确行为 | 用户或外部环境可见 |

## 2. 因果链

三者关系为：

**Fault → Error → Failure**

解释：

1. 软件中先存在静态缺陷，即 Fault；
2. 当测试或运行触发该缺陷时，程序内部状态可能变错，即 Error；
3. 如果错误状态继续传播到输出或外部行为，就产生 Failure。

注意：

- Fault 不一定导致 Error；
- Error 不一定导致 Failure；
- 测试的关键是让 Fault 被执行、让 Error 产生，并让 Error 传播到可观察输出。

类比：

##### **1. 核心定义与关系**

**软件故障（Fault）**  

相当于软件中的“先天缺陷”，是静态存在于代码、设计或文档中的偏差。例如：需求遗漏、逻辑错误、未处理的异常分支等。  

**类比**：医生发现的“根本病因”（如基因缺陷或器官结构异常），可能潜伏多年未被激活。

**软件错误（Error）**  

是软件运行时因故障被激活导致的**内部状态异常**。例如：变量值越界、内存溢出、信号逻辑错误等。  

**类比**：医生检测到的异常指标（如高血压、细菌感染），代表系统内部已偏离正常状态。

**软件失效（Failure）**  

是错误未被处理时，软件**外部可感知的功能丧失**。例如：系统崩溃、功能输出错误、用户请求超时等。  

**类比**：患者表现出的症状（如疼痛、器官衰竭），是内部问题传递到外部的结果。



##### **2. 因果链与动态过程**

三者遵循 **Fault → Error → Failure** 的传导路径：  

- **Fault**（缺陷存在）→ 被特定条件激活 → **Error**（内部异常）→ 未及时修复/容错 → **Failure**（外部失效）  

例如：  

- **Fault**：代码未处理除零异常 → **Error**：运行时变量值变为无穷大 → **Failure**：系统返回错误结果或崩溃

**关键区别**：  

- **软件故障是静态的**，可能长期存在而不引发失效（如冗余系统中的未激活缺陷）；  
- **硬件故障可能是动态的**（如元件老化、物理磨损），但软件故障仅由设计/实现时的错误导致。



**差异点**：  

- 软件故障不会“随时间退化”，而是从开发阶段就存在；  
- 硬件故障可能因物理老化（类似人体器官衰老），但软件失效仅由激活的缺陷引起。



##### **总结**

- **Fault是根源**，需通过代码审查、静态分析预防；  
- **Error是中间态**，需依赖运行时监控和容错机制拦截；  
- **Failure是终点**，需通过用户反馈和测试回溯修复。  

三者关系类似于疾病从病因到症状的传导，但软件缺陷的“先天固化性”使其与硬件/生物系统的动态退化本质不同。

## 3. 示例：numZero

```java
public static int numZero (int[] arr) {
    int count = 0;
    // Fault: 应从 i = 0 开始，而不是 i = 1
    for (int i = 1; i < arr.length; i++) {
        if (arr[i] == 0) {
            count++;
        }
    }
    return count;
}
```

### 测试用例 1

输入：`[2, 7, 0]`  
预期输出：`1`  
实际输出：`1`

分析：

- 执行了 Fault；
- 内部状态错误：循环从 `i = 1` 开始；
- 但最终输出仍正确；
- 因此没有 Failure。

### 测试用例 2

输入：`[0, 2, 7, 0]`  
预期输出：`2`  
实际输出：`1`

分析：

- 执行了 Fault；
- 内部状态错误；
- 错误传播到 `count`；
- 返回结果错误，产生 Failure。

# 三、PIE 模型

PIE 模型用于解释一个测试为什么能或不能发现 Fault。

| 阶段 | 含义 | 说明 |
|---|---|---|
| Execution / Reachability | 执行 / 可达 | 测试必须执行到含 Fault 的代码位置（A test may not execute the location of the fault !） |
| Infection | 感染 | Fault 必须导致程序内部状态错误（A test executing the fault may not produce an error !） |
| Propagation | 传播 | 错误状态必须传播到外部输出（An error may not be propagated to the output !） |

因此，一个测试用例发现 Failure 需要满足：

**执行 Fault → 产生 Error → Error 传播为 Failure**

## 示例：computeMean

```java
public static double computeMean (int[] arr) {
    int length = arr.length - 1; // Fault: 应为 arr.length
    double mean, sum;
    sum = 0.0;

    for (int i = 0; i < length; i++) {
        sum += arr[i];
    }
    mean = sum / (double) length;
    return mean;
}
```

| Case | 输入 | Fault | Error | Failure | 说明 |
|---|---|---|---|---|---|
| 1 | `[-3, 3, 0]` | 有 | 无 | 无 | 虽然 length 错，但 sum 和 mean 都正确 |
| 2 | `[3, 5, 4]` | 有 | 有 | 无 | sum 错，但 mean 仍等于正确结果 |
| 3 | `[3, 4, 5]` | 有 | 有 | 有 | sum 错，mean 也错 |

# 四、Verification 与 Validation

## 1. Verification：验证

**Verification**：判断某阶段的软件产品是否满足上一阶段建立的需求或规格。

核心问题：

> Are we building the product right?  
>
> 我们是否正确地构建了产品？

特点：

- 过程导向；
- 关注是否符合规格、设计、标准；
- 常见形式包括代码审查、设计评审、单元测试、静态分析等。

## 2. Validation：确认

**Validation**：在软件开发结束或阶段性完成后，评估软件是否符合预期用途。

核心问题：

> Are we building the right product?  
>
> 我们构建的是正确的产品吗？

特点：

- 结果导向；
- 关注是否满足真实用户需求；
- 常见形式包括验收测试、用户测试、真实场景测试等。

## 3. 对比表

| 维度 | Verification | Validation |
|---|---|---|
| 中文 | 验证 | 确认 |
| 关注点 | 过程是否正确 | 结果是否正确 |
| 核心问题 | 是否正确地构建产品 | 是否构建了正确的产品 |
| 依据 | 规格、设计、上一阶段要求 | 用户需求、实际用途 |
| 常见活动 | 审查、静态分析、单元测试 | 验收测试、用户测试 |

记忆：

- **Verification：过程导向**；
- **Validation：结果导向**。

# 五、规格说明与 Bug 的关系

**🌟 Specifications / 规范**

- Specifications play a key role.  / 规范起着关键作用

- Detailed specifications provide the correct behavior of the software.  / 详细规范提供了软件的正确行为

- They must describe normal and error behavior. / 它们必须描述正常和错误行为



规格说明用于描述软件的正确行为，包括：

- 正常行为；
- 异常行为；
- 输入输出关系；
- 约束条件。

软件可能被认为存在 Bug 的情况：

1. 规格要求做的事情，软件没有做；
2. 规格禁止做的事情，软件做了；
3. 软件做了规格没有提到的事情；
4. 规格没有写明但按常理应该做的事情，软件没有做；
5. 软件难以理解、难以使用、运行过慢等。

# 六、软件测试公理

![image-20260508153340410](https://cdn.jsdelivr.net/gh/BomLook/blog-pic@main/img/image-20260508153340410.png)

核心思想：**穷尽测试通常不现实。**

原因：

- 输入空间太大；
- 输出空间太大；
- 程序路径太多；
- 软件行为定义可能开放或模糊。

因此，测试的现实目标不是“证明没有 Bug”，而是：

- 在成本可控的情况下尽量降低风险；
- 尽早发现重要缺陷；
- 让缺陷尽快被修复。

![image-20260508153207459](https://cdn.jsdelivr.net/gh/BomLook/blog-pic@main/img/image-20260508153207459.png)

重要观点：

- 未发现的 Bug 称为 latent bugs，即潜在错误；
- 测试可能发现 Bug，但不能证明程序完全正确；
- Bug 是否修复还要考虑时间、成本、风险和需求定义是否正确（如果程序员在某个点粗心，那么很有可能在一片代码都会粗心犯错，所以 bug 很有可能只是冰山一角，需要持续的写新的、不同的测试用例）；
- “杀虫剂效应”说明重复使用同一类测试会逐渐失效，需要不断设计新的测试用例。

---

# 七、测试人员的目标

测试人员的主要目标：

1. 发现 Bug；
2. 尽早发现 Bug；
3. 确保 Bug 被修复。

注意：

- 测试人员的目标不是消除所有 Bug，因为现实中通常无法做到；
- 更现实的目标是以有限资源最大程度降低质量风险。

# Ch2 Test Process 测试流程

## 1. Waterfall Model：瀑布模型

![image-20260508153450419](https://cdn.jsdelivr.net/gh/BomLook/blog-pic@main/img/image-20260508153450419.png)

特点：

- 阶段顺序固定；
- 计划通常在早期完成；
- 后续阶段之间没有重叠；
- 测试通常在开发后期才开始。

### 优点

- 如果**早期需求和设计非常正确，后续返工较少**；
- 文档完整，便于团队成员查阅和交接。

### 缺点

- 项目前期缺少可见进展；
- 不灵活，难以应对需求变化；
- 文档工作耗时；
- 测试过晚，可能受时间和预算限制影响；
- 整体测试可能导致覆盖不充分；
- 如果测试发现需要重新设计的问题，修复成本高，可能被忽略；
- 用户不满意时，可能进入漫长维护阶段。

## 2. Spiral Model：螺旋模型

![image-20260508153902923](https://cdn.jsdelivr.net/gh/BomLook/blog-pic@main/img/image-20260508153902923.png)

特点：

- 风险驱动；
- 结合瀑布模型和快速原型迭代；
- 每一轮迭代包括目标确定、风险识别、开发与测试、计划下一轮。

### 优点

- 有利于风险管理；
- 可根据反馈逐步调整功能；
- 成本估计相对容易；
- 开发速度较快；
- 有用户反馈空间。

### 缺点

- 管理要求高；
- 文档更多；
- 可能难以按时或按预算完成；
- 更适合大型项目，小项目使用成本偏高。

## 3. V Model：V 模型

![image-20260508153952444](https://cdn.jsdelivr.net/gh/BomLook/blog-pic@main/img/image-20260508153952444.png)

![image-20260516162521016](https://cdn.jsdelivr.net/gh/BomLook/blog-pic@main/img/image-20260516162521016.png)

特点：

- 是瀑布模型的扩展；
- 强调开发阶段与测试阶段的对应关系；
- 体现 Verification 与 Validation；
- 编码完成后，测试从单元测试开始，逐步上升到集成测试、系统测试、验收测试。

### 优点

- 简单、结构清晰、易管理；
- 每个阶段都有可交付成果和评审；
- 强调各阶段的验证与确认；
- 提高测试在开发生命周期中的地位。

### 缺点

- 和瀑布模型类似，工作软件出现较晚；
- 不适合需求频繁变化的项目；
- 测试、调试和变更之间的关系表达不够清晰。

## 4. W Model：W 模型

![image-20260508154032097](https://cdn.jsdelivr.net/gh/BomLook/blog-pic@main/img/image-20260508154032097.png)

![image-20260516190400982](https://cdn.jsdelivr.net/gh/BomLook/blog-pic@main/img/image-20260516190400982.png)

特点：

- 也称 V 模型扩展或双 V 模型；
- 测试不再等到编码完成后才开始；
- 测试与开发并行；
- 强调开发人员和测试人员协作；
- 测试活动包括构建、执行和评估。

核心意义：

> W 模型体现了测试左移思想：越早开始测试活动，越容易降低后期修复成本。

## 5. Agile / XP：敏捷与极限编程

敏捷开发：强调**在短时间内构建可发布的软件。其时间周期以周为单位而非数月，且工作以高度协作的方式进行**。

渐进式开发：从**软件系统部分的简单实现开始。每一次增量，产品不断演进，不断添加增强功能，直到最终版本的实现**。测试是增量模型的重要组成部分，通常在每次迭代结束时进行。这意味着测试会在开发过程中更早开始，整体测试内容也更多。大部分测试采用回归测试形式，且可以大量重复使用测试用例和早期增量的测试数据。

- 优点：产品以更小的部分编写和测试，降低风险并方便纳入变更。客户或用户从一开始就参与其中，这意味着系统更有可能满足他们的需求，他们自己也更投入系统。
- 缺点：由于文档不足，与其他模型相比，管理起来可能比较困难。软件不断变化，随着体积增长，维护起来变得困难。

![image-20260508154528397](https://cdn.jsdelivr.net/gh/BomLook/blog-pic@main/img/image-20260508154528397.png)

XP 的核心理念：

- **迭代和增量开发；**
- **以用户价值为中心；**
- **拥抱需求变化；**
- **自组织团队；**
- **强调沟通、简单、反馈和勇气。**

XP 的价值：

- 沟通：极限编程的程序员与他们的客户和其他程序员交流；
- 简单：他们保持设计简单干净；
- 反馈：从一开始就通过软件测试获得反馈；
- 勇气：尽早将系统交付给客户，按照建议实施变更，勇于响应不断变化的需求

XP 的测试思想：

- 生命周期测试；
- 左移测试：从需求阶段就开始考虑测试；
- 持续测试：测试集成到每次迭代中，保证每次增量发布质量。

# 八、TDD：测试驱动开发

## 1. 定义

**TDD（Test-Driven Development）**：在编写功能代码之前先编写测试。**帮助开发者思考接口设计以及边界条件**

基本流程：

1. 编写测试；
2. 编译；
3. 修复编译错误；
4. 运行测试并观察失败；
5. 编写刚好能通过测试的代码；
6. 运行测试并观察通过；
7. 重构代码；
8. 重复以上过程。

![image-20260508155236045](https://cdn.jsdelivr.net/gh/BomLook/blog-pic@main/img/image-20260508155236045.png)

## 2. 实践原则

- 只有当自动化测试失败时才编写功能代码；
- **如果通过其他方式发现 Bug，应先写一个失败测试，再修复 Bug**；
- 尽可能频繁运行测试，最好每次代码变更后都运行；
- **单元测试让重构更安全**；
- 缺少单元测试时，代码变更更容易破坏已有功能。

## 3. 好处

- **单元测试会被实际编写出来；**
- 通过测试**明确接口和行为**；
- 提供可证明、可重复、自动化的验证；
- 增强开发者重构代码的信心；
- 有助于提高代码可维护性。

# 九、验收测试、CI 与测试金字塔

## 1. 验收测试

验收测试应与用户故事对齐：

- 为每个用户故事定义验收标准；
- 根据验收标准设计测试；
- 判断软件是否满足用户预期。

## 2. CI：持续集成

持续集成通过每日构建和自动化测试提供持续反馈。

作用：

- **快速发现集成问题；**
- **提高反馈速度；**
- **支持持续改进。**

## 3. 测试金字塔

测试金字塔从下到上为：

1. **单元测试**：数量最多，速度快，成本低，定位问题方便；
2. **集成 / API 测试**：验证模块之间的交互；
3. **端到端 / UI 测试**：数量较少，成本高，但能覆盖关键用户流程。

核心思想：

> 单元测试应构成自动化测试的主体，UI 测试应少而关键。

# Ch3 Test Principle 测试原理

## 1. 静态验证与动态验证

| 维度 | 静态验证 | 动态验证 |
|---|---|---|
| 是否执行代码 | 不执行 | 执行 |
| 主要方式 | 代码阅读、静态分析、代码审查、编码规范检查，可以采用数学方法，包括对程序进行符号执行，可以是一种形式化的方法，包括规范与源代码之间转换的符号验证。 | 运行测试用例 |
| 关注内容 | 代码、文档、规范之间的一致性 | 输入、实际输出、预期输出之间的一致性 |

动态验证的基本过程：

1. 设计测试用例；
2. 选择测试数据；
3. 确定预期输出；
4. 执行程序；
5. 比较实际输出与预期输出。

## 2. 黑盒测试与白盒测试

| 维度 | 黑盒测试 | 白盒测试 |
|---|---|---|
| 依据 | 需求说明 / 规格说明 | 源代码实现 + 规格说明 |
| 是否需要代码 | 不需要 | 需要 |
| 代码修改后测试用例是否可复用 | 通常可以 | 通常不可以或需要调整 |
| 自动化难度 | 相对较难 | 相对较易 |
| 擅长发现 | 功能是否符合需求 | 代码结构、路径、分支、逻辑问题 |

重要区别：

- 白盒测试不擅长发现“遗漏功能”，因为代码中没有实现的功能无法通过代码结构直接发现；
- 黑盒测试不擅长发现“额外功能”，因为规格中未定义的内部行为可能难以被覆盖。



黑箱测试的基本原理可以用多种方式表达：

1. 与规范进行测试。
2. 根据规范使用测试覆盖标准。
3. 开发基于规范的测试用例。
4. “演练”规范。



白盒测试的基本原理可以表达为：

1. 对实现进行测试
2. 使用基于实现的测试覆盖标准
3. 开发基于实现的测试用例
4. “演习”实施

# 十一、最终压缩版记忆框架

可以用以下框架复习：

**测试是什么？**  

判断质量，发现错误。

**为什么要测试？**  

软件复杂、需求模糊、资源有限，质量风险高。

**测试发现的是什么？**  

Fault、Error、Failure。

**为什么有些 Fault 测不出来？**  

因为必须满足 PIE：执行、感染、传播。

**怎么判断软件对不对？**  

依据规格说明、Verification 和 Validation。

**测试在流程中什么时候做？**  

从瀑布的后置测试，发展到 V / W / Agile / TDD / CI 的持续测试。

**测试有哪些基本类型？**  

静态 vs 动态；黑盒 vs 白盒。
