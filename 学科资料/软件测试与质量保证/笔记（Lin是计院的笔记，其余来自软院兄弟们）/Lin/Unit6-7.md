## ✅ **第六章：性能测试（重要概念）**

根据重点回顾，本章要求掌握“**性能测试中的重要概念**”。考试中可能会考查概念的定义、区别以及在实际测试中的应用。

### 一、核心概念（必须掌握）

| 概念 | 定义 | 关键点 |
|------|------|----------|
| **性能测试** | 检测系统在特定情况下的响应能力和稳定性，评价是否与性能需求一致 | 关注“是否达到需求”，超过需求不是重点 |
| **负载测试** | 通过**逐渐增加**负载（如并发用户数、事务数），观察系统在不同负载下的表现 | 负载是渐进的，找出性能拐点 |
| **压力测试** | 评估系统处于或**超过预期负载**时的运行情况，发现系统崩溃的临界点 | 关注峰值或超载下的行为，系统不应崩溃 |
| **响应时间** | 从服务器收到请求到完成执行的时间间隔 | 端到端、网络、服务器、中间件等多种度量方式 |
| **并发量** | 在极短时间（毫秒级）内同时发出的请求数量 | 影响系统处理混乱的能力 |
| **吞吐量** | 单位时间内完成的工作量（如交易数/秒） | 衡量系统处理能力的最直接指标 |

### 二、性能度量方法

- **系统资源开销**：CPU、内存、磁盘I/O、网络流量
- **稳定性**：是否容易宕机
- **执行速度**：响应时间、吞吐量、吞吐率

### 三、性能测试、负载测试、压力测试的区别（重点辨析）

- **性能测试**：是否满足性能需求（如支持5000并发）
- **负载测试**：逐渐加负载，观察变化趋势（找到拐点）
- **压力测试**：超过预期负载，测试极限和崩溃点
> 注意：实际工作中“性能测试”常作为三者统称，但考试需区分定义。

### 四、影响响应时间、并发量、吞吐量的因素

- **响应时间**：交易类型、交易组合、并发数、服务器配置、距离
- **并发量**：事务资源消耗、硬件配置、系统配置、代码质量
- **吞吐量**：同上，且与并发量、响应时间密切相关

### 五、性能测试流程（PPT中最后部分，可作为补充了解）

- 产生需求与目标 → 设计方案 → 设计用例 → 搭建环境 → 开发脚本 → 执行测试 → 数据分析 → 性能对比


## ✅ **第七章：网络安全性测试（考试重点）**

### 一、常见的网络攻击方法（PPT中详细列举）

| 攻击方法                                         | 具体说明 / 案例                                                                                        |
| -------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| **Google Hack**                              | 利用Google搜索漏洞或敏感信息，如`intitle:"Index of" passwd`、`filetype:pdf confidential`；Santy蠕虫利用Google搜索自动传播 |
| **直接对象引用（Insecure Direct Object Reference）** | 修改URL中的参数（如`file=../../../../etc/passwd`）访问未授权文件；或修改`userid`查看他人账户                               |
| **恶意代码执行（Malicious File Execution）**         | 上传WebShell或木马，如dvbbs论坛空字节注入漏洞：将`web.jpg`改名为`web.asp`+空字节，使服务器保存为可执行的`.asp`文件                     |
| **注入攻击（Injection）**                          | 主要是SQL注入：字符串参数注入（`' OR 1=1 --`）、数字参数注入（`union select`）、盲注入（利用`and exists`二分法猜解）                  |
| **跨站脚本攻击（XSS）**                              | 存储式XSS（恶意代码保存在服务器，如论坛发帖`<script>alert(1)</script>`）；反射式XSS（恶意链接，如邮件中的图片标签触发转账）                   |
| **跨站请求伪造（CSRF）**                             | 利用用户已登录的身份发起恶意请求，如`<img src="http://bank.com/transfer?to=Alice&amount=100">`                     |
| **暴力猜解**                                     | 自动尝试用户名/密码，配合字典攻击                                                                                |
| **网络窃听（Sniffer）**                            | 捕获网络中传输的明文密码、Cookie等                                                                             |
| **错误信息利用**                                   | 利用错误页面泄露的服务器版本、数据库类型、SQL语句、文件路径等                                                                 |
| **URL访问控制不当**                                | 未统一权限控制，导致未登录可直接访问后台页面                                                                           |
| **不安全的存储**                                   | 明文保存用户口令，一旦数据库泄露即可获取                                                                             |

---


### 二、网络攻击防范措施（针对每种攻击）

| 攻击类型        | 防范措施（PPT中明确给出）                                                                                                                      |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| **直接对象引用**  | 统一权限控制框架；对每个资源访问进行鉴权；避免使用可预测的ID（如使用UUID）                                                                                            |
| **恶意代码上传**  | ①白名单检查扩展名（只允许`jpg,gif`等）；②过滤特殊字符（`../`、`\0`空字节）；③上传目录设置为不可执行脚本；④对文件名重命名（如时间戳+随机数）                                                   |
| **SQL注入**   | ①输入验证（数字强转、字符串转义）；②使用参数化查询（JSP的`PreparedStatement`、ASP.NET的`SqlParameterCollection`）；③最小权限原则（不用`sa`或`dba`）；④禁用敏感存储过程（`xp_cmdshell`） |
| **XSS**     | ①输入过滤（去除`<script>`、`onclick`等标签/事件）；②输出编码（将`<>&`编码为`&lt;&gt;&amp;`）；③输出过滤（允许少量标签时用白名单）；④用户端加固（禁用ActiveX、不点可疑链接、不开启自动登录）             |
| **CSRF**    | ①使用请求令牌（Token）；②验证Referer头；③敏感操作不使用GET方法；④重要操作二次确认                                                                                  |
| **信息泄露**    | 统一异常处理，错误提示不暴露具体信息（如登录失败统一提示“用户名或密码错误”）                                                                                             |
| **暴力破解**    | 图片验证码；短信/邮箱二次验证；限制失败尝试次数；双因素认证                                                                                                      |
| **网络窃听**    | 通信加密（HTTPS、TLS）；避免明文传输敏感信息                                                                                                          |
| **不安全的存储**  | 使用安全哈希算法（如bcrypt、PBKDF2）加盐存储密码                                                                                                      |
| **URL访问控制** | 建立统一的权限控制框架（如拦截器、过滤器），所有页面均需鉴权                                                                                                      |
|             |                                                                                                                                     |

### 三、网络安全性测试的总体思路（PPT中的步骤总结）

1. **信息收集**：
   - 使用Google Hack搜索敏感信息
   - 网页爬行，获取所有URL、参数、表单
   - 暴力猜解目录、文件名、参数
   - 使用漏洞扫描器（如WebScarab、OWASP ZAP）

2. **分析攻击面**：
   - 检查所有输入点：URL参数、POST数据、Cookie、HTTP头（Referer、User-Agent等）
   - 检查文件上传、下载功能
   - 检查业务逻辑中的权限、状态转换

3. **构造恶意输入**：
   - SQL注入（`' OR 1=1 --`、`union select`）
   - XSS（`<script>alert(1)</script>`）
   - 命令注入（`; rm -rf /`）
   - 路径遍历（`../../etc/passwd`）

4. **利用错误信息**：
   - 触发错误，查看是否泄露数据库类型、版本、路径、SQL语句

5. **测试权限控制**：
   - 直接对象引用：修改ID、文件名参数，尝试越权访问
   - 未授权访问：直接访问后台URL

6. **测试文件上传**：
   - 上传恶意文件（如`shell.jsp`、`web.asp`）
   - 空字节注入（`file.php\0.jpg`）

7. **测试业务逻辑**：
   - 绕过流程（如跳过支付直接确认订单）
   - 重放攻击、并发竞争

8. **使用工具辅助**：
   - WebScarab（HTTP代理、拦截、修改）
   - OWASP工具集

### 四、蠕虫、木马、攻防方法

| 概念 | PPT中的定义/案例 | 关键点 |
|------|------------------|--------|
| **蠕虫（Worm）** | Samy蠕虫：利用MySpace的XSS漏洞，通过CSRF方式传播，指数级增长；Santy蠕虫：利用phpBB漏洞上传perl脚本，通过Google搜索新目标自动传播 | 能**自我复制**、**自动传播**，无需用户干预 |
| **木马（Trojan）** | WebShell（如dvbbs漏洞上传的`web.asp`）是一种木马，伪装成正常文件（如头像图片），实为远程控制后门 | 需要**诱骗用户执行**，通常伪装成正常程序 |
| **攻防方法** | **攻击方法**：SQL注入、XSS、CSRF、文件上传、路径遍历、暴力破解等；**防御方法**：安全开发生命周期（安全编码、输入验证、输出编码、权限控制、配置加固、安全测试、使用防火墙/IDS/IPS） | 攻击重在“构造恶意输入”，防御重在“不信任用户输入” |
