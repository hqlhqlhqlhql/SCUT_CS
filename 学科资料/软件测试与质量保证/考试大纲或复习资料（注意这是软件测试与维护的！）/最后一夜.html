软件测试基础概念

Software / Bug / Fault-Error-Failure / PIE / V&V / Specification

Software 组成

指令数据结构配置文件系统文档用户文档

软件不只是代码。系统文档描述系统结构，用户文档解释系统如何使用。

Bug

Bug 是非正式术语，不是精确定义。精确术语中应区分 Fault、Error 和 Failure。

Fault / Error / Failure

Fault
静态缺陷→Error
内部错误状态→Failure
外部错误行为

代码中有 Fault 不一定会立刻产生 Failure。只有 Fault 被执行、造成错误状态，并且错误状态传播到外部输出时，才会观察到 Failure。

PIE Model

Fault 变成 Failure 的三个条件
Execution / Reachability
执行到 Fault 所在代码→Infection
内部状态变错→Propagation
传播到输出

Verification vs Validation

概念关注点经典问题
Verification某阶段产品是否满足上一阶段需求是否正确地构建产品？
Validation最终软件是否符合预期用途是否构建了正确的产品？

Specification 与 Software Testing

Specification 提供软件的正确行为，是判断 expected output 的依据，必须描述正常行为和错误行为。

Software Testing 是开发高质量软件系统的关键元素，是判断质量和发现 Bug 的系统化方法，不只是运行程序找 Bug。

本章中文背诵版

软件不只是代码。一个软件系统通常包括指令、数据结构、配置文件、系统文档和用户文档。

Bug 是非正式术语。在精确的软件测试术语中，应区分 Fault、Error 和 Failure。Fault 是软件中的静态缺陷，Error 是由 Fault 引起的内部错误状态，Failure 是相对于需求或预期行为而言的外部错误行为。三者关系是 Fault → Error → Failure。

PIE 模型说明 Failure 出现的条件：必须执行到 Fault 所在代码，Fault 必须导致内部状态出错，错误状态还必须传播到输出。只有这三个条件都满足，Fault 才会表现为 Failure。

Verification 关注某阶段产品是否满足上一阶段建立的要求，问的是“是否正确地构建产品”。Validation 关注最终软件是否符合预期用途，问的是“是否构建了正确的产品”。

Specification 在测试中起关键作用，因为它提供软件的正确行为，必须描述正常行为和错误行为。软件测试是判断质量和发现 Bug 的系统化方法，不只是运行程序找 Bug。

软件测试公理与测试目标

Nine Axioms / Risk-based / Tester Goals

九条公理

  1. 不可能完全测试一个程序。
  2. 软件测试是一种基于风险的活动。
  3. 测试不能证明 Bug 不存在。
  4. 发现的 Bug 越多，说明可能还有更多 Bug。
  5. 并非所有发现的 Bug 都会被修复。
  6. 很难判断什么时候一个 Bug 真的是 Bug。
  7. 规格说明永远不是最终版。
  8. 软件测试人员不是项目中最受欢迎的成员。
  9. 软件测试是一门有纪律性和技术性的职业。

无法完全测试

输入、输出和程序路径数量都很大，并且规格说明存在解释空间。

基于风险

不能测试所有输入，只能跳过一些输入；跳过输入意味着承担风险。

不能证明无 Bug

测试通过只能说明在给定条件下没有发现 Bug 或没有观察到 Failure。

测试人员目标

发现 Bug→尽早发现 Bug→确保 Bug 被修复

本章中文背诵版

软件测试有九条公理：不可能完全测试一个程序；软件测试是基于风险的活动；测试不能证明 Bug 不存在；发现的 Bug 越多，说明可能还有更多 Bug；并非所有 Bug 都会被修复；很难判断什么时候一个 Bug 真的是 Bug；规格说明永远不是最终版；测试人员不是项目中最受欢迎的成员；软件测试是一门有纪律性和技术性的职业。

不可能完全测试程序，是因为输入、输出和程序路径数量都很大，并且规格说明存在解释空间。由于不能测试所有输入，测试人员必须跳过一些输入，这就意味着承担风险。

测试可以证明 Bug 存在，但不能证明 Bug 不存在。测试通过只能说明在给定测试条件下没有发现 Bug 或没有观察到 Failure。测试的实际价值是降低风险，而不是证明软件没有 Bug。

软件测试人员的目标是发现 Bug、尽早发现 Bug，并确保 Bug 被修复。

静态 / 动态验证与黑盒 / 白盒测试

Static / Dynamic Verification · Black-box / White-box
类型是否执行代码说明
Static Verification不执行阅读代码、静态分析、符号执行、形式化验证
Dynamic Verification执行通过执行程序确认运行行为，也就是软件测试

Dynamic Verification 流程

测试用例→测试数据→输入值→实际输出→预期输出→比较
测试方法依据关注点覆盖
Black-box TestingSpecification外部行为、输入输出、需求满足覆盖 Specification，但不完整覆盖 Implementation
White-box TestingImplementation / Code Structure代码结构、路径、分支、条件、内部逻辑覆盖 Implementation，但不覆盖 Specification

Omission / Commission

Omission

该有的没有。Specification 要求有，但 Implementation 没实现。白盒测试容易漏掉。

Commission

多做了不该做的。Implementation 实现了 Specification 没要求的额外行为。黑盒测试容易漏掉。

本章中文背诵版

静态验证不需要执行代码，动态验证需要执行代码。动态验证就是软件测试，通过执行程序来确认程序运行行为。动态验证过程中，测试用例指导选择测试数据，测试数据包括输入值和预期输出值，输入值被提供给程序，程序产生实际输出，再与预期输出进行比较。

黑盒测试完全基于 Specification，关注外部行为和需求是否满足，不需要了解内部代码结构。白盒测试基于 Implementation 和代码结构，关注路径、分支、条件和内部逻辑。

黑盒测试覆盖 Specification，但不完整覆盖 Implementation；白盒测试覆盖 Implementation，但不覆盖 Specification。黑盒可能漏掉额外实现的不该有功能，即 Commission errors；白盒可能漏掉根本没有实现的应有功能，即 Omission errors。

软件过程模型

Waterfall / Spiral / V / W / Agile / TDD
Waterfall顺序式开发；开始时完成所有计划；阶段不重叠；测试通常靠后。
Spiral风险驱动；结合瀑布与快速原型；从设计目标开始，以客户评审结束。
V ModelWaterfall 扩展；强调 V&V；开发阶段和测试活动对应。
W Model测试与开发并行；测试不是等代码实现后才开始。
Agile测试人员参与迭代需求协商，并立即转成测试用例。
TDD测试优先；先写单元测试，再写代码让测试通过，最后重构。

Spiral 流程

确定目标→识别风险→开发和测试→计划下一轮迭代

V Model 文档对应

文档对应测试
用户需求规格说明验收测试
系统需求规格说明系统测试
系统设计规格说明集成测试
详细设计规格说明单元测试

TDD 流程

开始→为新功能写测试→编译→修复编译错误→运行测试并失败→写代码→运行测试并通过→按需重构

本章中文背诵版

Waterfall 是顺序式开发模型，所有计划在开始时完成，各阶段没有重叠，测试通常在后期进行。它的优点是早期需求和设计如果正确，可以节省后期成本，并且文档集中；缺点是对变更不灵活，测试靠后，文档耗时，项目后期才有明显进展，整体测试可能不完整。

Spiral 是风险驱动的开发过程，结合瀑布模型和快速原型迭代模型。它从设计目标开始，以客户评审进展结束。流程是确定目标、识别风险、开发和测试、计划下一轮迭代。

V Model 是 Waterfall 的扩展，强调 Verification 和 Validation，把开发阶段和测试活动对应起来。实现完成后，从单元测试开始，逐级到集成测试、系统测试和验收测试。文档对应关系是：用户需求对应验收测试，系统需求对应系统测试，系统设计对应集成测试，详细设计对应单元测试。

W Model 强调测试不是在代码实现后才开始，而是和开发过程并行进行，并强调开发与测试协作。Agile 中测试人员从每次迭代的需求协商阶段就参与，并立即把确认的需求转成测试用例。TDD 是测试优先开发，先写单元测试，再写代码让测试通过，最后按需重构。

单元测试与 Scaffolding

Unit Testing / Driver / Stub

Unit Testing 特点

  • 主要是白盒测试
  • 由开发者编写
  • 验证小单元程序功能
  • 需要完整了解实现细节
  • 隔离测试一个软件片段
  • 确保“正确地构建软件”

Unit Testing 作用

  • 定位错误
  • 尽早发现错误
  • 避免不必要功能
  • 提高代码质量
  • 集成前确保单元最低质量

Unit Testing 流程

计划→编写测试用例→执行测试→分析结果→修复并回归→自动化与 CI/CD→维护用例

Scaffolding 结构

Driver
模拟上层调用者→Unit
被测对象→Stub
模拟下层依赖

本章中文背诵版

单元测试主要是开发者编写的白盒测试，用于验证程序功能的小单元。它需要完整了解实现细节，并且通常与实现同步进行。单元测试的目标是确保开发者正确地构建软件。

单元测试的作用包括定位错误、尽早发现错误、避免不必要功能、提高代码质量，并在集成前确保单元最低质量。单元测试主要关注独立路径、数据结构、接口、边界值、错误处理和内存。

Scaffolding 是测试支持代码，用来模拟外部代码，使单元可以被隔离测试。Driver 模拟上层调用者，调用被测软件并传入测试数据；Stub 模拟下层依赖服务，提供固定或有限的一组值。调用关系是 Driver → Unit → Stub。

集成测试

Integration Testing Strategies

集成测试用于验证系统中的组件或子系统是否能够正确协同工作，关注接口和交互。

级别关注点
Unit Testing单个小单元，隔离，白盒为主
Integration Testing组件或子系统之间的接口和交互
System Testing整个系统是否满足需求和场景

四种集成策略

Big Bang

一次性集成所有模块，简单但故障定位困难。

Top-Down

从主程序向下集成，主要使用 Stubs，可以较早形成系统原型。

Bottom-Up

从低层模块向上集成，主要使用 Drivers，主程序最后测试。

Sandwich

结合 Top-Down 和 Bottom-Up，上下并行，向中间收敛。

本章中文背诵版

集成测试用于验证系统中的组件或子系统是否能够正确协同工作。它关注组件之间的接口和交互，并确认交互后产生正确结果。单元测试通过不代表集成测试一定通过，因为单个单元独立正确，组合后仍可能出现参数、接口、数据格式、调用顺序或单位不一致等集成故障。

Big Bang 一次性集成所有模块，简单但故障定位困难。Top-Down 从主程序向下集成，主要使用 Stubs，可以较早形成系统原型。Bottom-Up 从低层模块向上集成，主要使用 Drivers，主程序最后测试。Sandwich 结合 Top-Down 和 Bottom-Up，上下并行并向中间收敛，同时使用 Stubs 和 Drivers。

系统测试

System / Functional / Non-functional / Regression

System Testing

测试系统作为整体在场景和需求方面的行为；假设组件已正常工作；使用黑盒测试技术。

Functional Testing

验证软件行为是否符合预期，也测试坏输入以检查隐含假设。

Regression Testing

代码变更后重新运行测试，确保已有行为没有被破坏。

功能测试策略

场景测试等价类划分边界值分析错误猜测决策表测试随机测试

本章中文背诵版

系统测试测试系统作为整体在场景和需求方面的行为。它通常在集成测试之后进行，假设所有组件已经正常工作，使用黑盒测试技术，并将软件与顶层需求进行核对。

功能测试验证软件行为是否符合预期，也测试坏输入，以检查系统的隐含假设。系统功能测试通过系统接口输入数据并收集结果，而不是像单元测试那样直接调用方法。

非功能测试测试软件质量而不是具体功能，也叫各种 “-ilities”，如易用性、可靠性、可维护性、安全性和性能。回归测试是在代码变更后重新运行测试，确保已有功能、性能、安全性等没有被破坏。

性能测试

Performance / Load / Stress / Endurance / Spike / Volume

常见性能问题

加载时间长响应时间差可扩展性差瓶颈

常见瓶颈

CPU内存网络磁盘

五类性能测试

类型含义
Load Testing测预期用户负载下的执行能力。
Stress Testing测极端负载下的表现和恢复能力，找崩溃点。
Endurance Testing测长时间预期负载。
Spike Testing测突然大负载峰值。
Volume Testing测大量数据下系统行为。

性能测试流程

识别测试环境→确定接受标准→计划设计→配置环境→实现设计→运行测试→分析调优重测

本章中文背诵版

性能测试用于检查软件在工作负载下的响应性和稳定性，目标是识别并移除性能瓶颈，并验证软件是否满足速度、可扩展性和稳定性要求。

常见性能问题包括加载时间长、响应时间差、可扩展性差和瓶颈。常见瓶颈包括 CPU、内存、网络和磁盘。常见指标包括响应时间、吞吐量、每秒请求数、错误率、CPU 使用率、内存使用率、磁盘使用情况、网络延迟和带宽使用率。

Load Testing 测预期用户负载下的执行能力；Stress Testing 测极端负载下的系统表现和恢复能力；Endurance Testing 测长时间预期负载；Spike Testing 测突然大负载峰值；Volume Testing 测大量数据下系统行为。

Performance Testing 是 Load 和 Stress 的超集，用于建立系统性能基准。Load 在预期负载或崩溃阈值以下，Stress 在超过崩溃阈值的极端负载下。性能测试流程是识别测试环境、确定性能接受标准、计划和设计性能测试、配置测试环境、实现测试设计、运行测试、分析调优并重测。

验收测试

Acceptance / Alpha / Beta / Four Levels

验收测试是软件测试的最终阶段，在系统测试之后、最终交付客户之前进行，由客户或最终用户执行，采用黑盒测试方法。

Software Configuration Review

检查交付的软件配置项和文档是否完整、规范、一致。

Software Validity Testing

检查软件在业务和运行环境下是否有效，包括接口、易用性、功能、性能、安全性等。

Alpha vs Beta

维度Alpha TestingBeta Testing
地点开发者场地客户或最终用户场地
环境受控环境真实环境
控制受开发者控制不受开发者控制
开放不对外开放向客户或公众开放
目的发布前发现 Bug收集真实反馈并改进产品

四大测试总结

测试测试对象测试依据测试人员测试方法测试数据
单元测试软件单元详细设计开发者白盒为主一般不用真实数据
集成测试模块接口和交互架构设计开发者和测试人员黑盒为主，白盒为辅一般不用真实数据
系统测试整个软硬件系统需求规格说明专业测试人员完全黑盒尽量使用或模拟真实数据
验收测试整个软硬件系统需求、合同、验收标准用户主导，开发者和测试人员协作完全黑盒尽量使用或模拟真实数据

本章中文背诵版

验收测试是软件测试的最终阶段，在系统测试之后、最终交付客户之前进行。它由客户或最终用户执行，用于检验系统是否符合业务需求、是否适合交付，并采用黑盒测试方法。

验收测试包括 Software Configuration Review 和 Software Validity Testing。前者检查交付物和文档是否完整、规范、一致；后者验证软件在业务和运行环境下是否有效，包括接口、易用性、功能、性能、环境、可靠性、兼容性、备份和安全性等。

Alpha Testing 在开发者场地、受控环境中进行，不对外开放，主要用于产品发布给外部用户前发现 Bug。Beta Testing 在 Alpha 之后进行，在客户或最终用户真实环境中进行，没有开发者控制，通常向客户或公众开放，依赖真实用户反馈改进产品。

四大测试中，Unit 测软件单元，开发者负责，白盒为主；Integration 测模块接口和交互，开发者和测试人员协作，黑盒为主白盒为辅；System 测整个系统，专业测试人员负责，完全黑盒；Acceptance 测整个系统，用户主导，完全黑盒。

自动化测试

Automation Testing

为什么需要自动化

手工测试慢、容易出错、难以准确重复；软件测试需要快速、准确和可重复，尤其适合回归测试。

自动化测试是什么

借助专门工具比较测试用例的预期结果和实际结果，可以自动执行、收集、评估、报告和测量覆盖率。

自动化测试流程

定义目标和策略→选择工具→搭建环境→开发脚本→执行并分析

优点

  • 高投资回报率
  • 更快上市
  • 重复测试用例
  • 大型测试矩阵
  • 并行执行
  • 无人值守
  • 减少人为错误
  • 节省时间和金钱

缺点

  • 工具通常昂贵
  • 不适合测试用户体验
  • 需要编码知识和经验

本章中文背诵版

自动化测试是借助专门的自动化测试工具，比较测试用例的预期结果和实际结果的软件测试方法。它可以自动执行测试或测试集合，自动收集和评估结果，自动生成报告，并自动测量测试覆盖率。

我们需要自动化测试，因为手工测试慢、容易出错、难以准确重复，而软件测试需要快速、准确和可重复，尤其适合回归测试。

自动化测试的优点包括高投资回报率、更快上市、支持重复测试用例、支持大型测试矩阵、支持并行执行、无人值守执行、提高准确性、减少人为错误，并节省时间和金钱。

自动化测试的缺点包括工具成本高、不适合测试用户体验，并且需要编码知识和经验。手工测试适合短期低预算项目、探索性测试、临时测试和用户体验测试；自动化测试适合回归测试、负载测试、压力测试、大型项目和性能测试。

软件维护

Software Maintenance / CAPP / Maintainability

软件维护是在软件交付或投入使用后对软件进行修改的过程，通常不涉及系统架构重大变化，需要程序理解。

CAPP 四类维护

Corrective

纠错性维护，修复潜在错误。

Adaptive

适应性维护，适应外部环境变化。

Perfective

完善性维护，改进已交付软件。

Preventative

预防性维护，提高未来可维护性。

主要维护活动

识别变更→管理过程→理解程序→做出修改→测试修改→文档化并记录

本章中文背诵版

软件维护是在软件交付或投入使用后对软件进行修改的过程。它通常不涉及系统架构的重大变化，而是通过修改已有组件或添加新组件实现，并且需要程序理解。

软件变化不可避免，因为使用中会出现新需求，业务环境会变化，故障必须修复，新硬件和设备会加入系统，性能或可靠性可能需要改进，并且软件与运行环境紧密耦合。

Corrective 是纠错性维护，用于修复潜在错误；Adaptive 是适应性维护，用于适应外部环境变化；Perfective 是完善性维护，用于改进已交付软件；Preventative 是预防性维护，用于提高未来可维护性。

Maintainability 是软件被修改的容易程度。Ripple Effect 是一个位置的变更可能影响其他组件。Impact Analysis 是识别某个变更会如何影响系统其他部分的过程。主要维护活动包括识别变更、管理过程、理解程序、做出修改、测试修改、文档化并记录修改。

提高可维护性的方法包括清晰写文档、记录假设、使用版本控制、清晰命名、保持代码整洁、遵守代码风格、保持代码简单，并避免过长方法和不必要复杂性。

静态测试与代码评审

Static Testing / Code Review

Static Testing

不执行软件，仔细且有条不紊地审查和分析软件以发现 Bug，包括 Code Review 和 Static Program Analysis。

价值

许多 Fault 可以不执行软件就被发现，更早修复，减少执行型测试时间，降低成本并提高生产率。

Code Review 四大元素

发现问题→遵守规则→提前准备→撰写报告
类型特点
Peer Review非正式小组评审，由程序员和 / 或测试人员作为 reviewers。
Walkthrough更正式，由代码作者逐行解释代码，reviewers 听并提问。
Formal Code Inspection更全面，有 inspectors 和 moderator，检查后形成报告，必要时重新检查。

Formal Code Inspection 流程

概览→准备→检查→修改→跟进

本章中文背诵版

静态测试是在不执行软件的情况下，仔细且有条不紊地审查和分析软件以发现 Bug 的过程，包括 Code Review 和 Static Program Analysis。它的价值在于许多 Fault 可以不执行软件就被发现，并且可以更早修复，从而减少执行型测试时间、降低成本并提高生产率。

Code Review 的四大元素是发现问题、遵守规则、提前准备和撰写报告。Peer Review 是非正式小组评审，由程序员和 / 或测试人员作为 reviewers；Walkthrough 更正式，由代码作者逐行解释代码，reviewers 听并提问。

Formal Code Inspection 比 Walkthrough 更全面，有 inspectors 和 moderator，检查后形成报告，程序员修改，必要时重新检查。它的流程是概览、准备、检查、修改和跟进。Static Program Analysis 使用工具在不执行程序的情况下分析源代码，可以发现缓冲区溢出、空指针、未初始化数据、资源泄漏、API 协议问题和数据竞争等问题。

Test Plan

描述软件测试范围和活动的文档，是正式测试软件或产品的基础，说明范围、方法、资源和进度安排。

定义测试策略→定义测试系统→估算测试工作量→准备并评审测试计划
文档作用
Bug Report说明软件具体出了什么问题，目标是让 Bug 被理解、复现并修复。
Test Summary Report测试结束后的重要交付成果，总结测试活动、结果、缺陷、风险和结论。

本章中文背诵版

Test Plan 是描述软件测试范围和活动的文档，是正式测试软件或产品的基础。它说明测试活动的范围、方法、资源和进度安排。Test Plan 可分为 Master Test Plan、Testing Level Specific Test Plans 和 Testing Type Specific Test Plans。

Test Planning Process 包括定义测试策略、定义测试系统、估算测试工作量、准备并评审测试计划。Test Strategy 要考虑方法、入口和出口标准、自动化、范围和优先级。

Test Case 是测试的最小单位，通常包括前置条件、输入、操作步骤、预期输出和后置条件。一个 Test Case 可以对应至少一个 Test Script，一组相关的 Test Cases 组成 Test Suite。

Bug Report 是说明软件具体出了什么问题的信息文档，目标是让 Bug 被理解、复现并修复。好的 Bug Report 应该清晰、可执行、容易完成。Test Summary Report 是测试结束后的重要交付成果，用于向项目干系人总结测试活动、结果、缺陷、风险和结论。
